## 🎯 Основы <a id="основы"></a>  

### Что такое Performance Testing <a id="что-такое-performance-testing"></a>

**Performance Testing (нагрузочное тестирование производительности)** — это вид нефункционального тестирования, целью которого является оценка поведения системы под нагрузкой: скорости, отзывчивости, стабильности и потребления ресурсов.

Простыми словами: performance testing отвечает на вопрос **«Как быстро и надёжно работает система, когда ею пользуются реальные пользователи?»**

#### Что проверяет Performance Testing:

| Аспект | Что оценивается |
|--------|----------------|
| **Скорость** | Время ответа (response time) на запросы пользователя |
| **Пропускная способность** | Сколько операций/запросов система обрабатывает в секунду (TPS/RPS) |
| **Стабильность** | Не падает ли производительность со временем |
| **Масштабируемость** | Как растёт производительность при добавлении ресурсов |
| **Надёжность под нагрузкой** | Не появляются ли ошибки при высоком количестве пользователей |

#### Пример из жизни:

> Вы разрабатываете интернет-магазин. Без нагрузки страница открывается за 0.3 секунды. Но в «чёрную пятницу» одновременно заходят 5000 пользователей. Performance testing покажет: упадёт ли время ответа до 5 секунд, появятся ли ошибки 500, или система выдержит достойно.

#### Чем performance testing НЕ является:

| Не путать с | Разница |
|-------------|---------|
| **Функциональным тестированием** | Там проверяют «работает ли кнопка?». Здесь — «как *быстро* она срабатывает под нагрузкой?» |
| **Unit-тестированием** | Unit-тесты проверяют методы изолированно. Performance test — систему в сборе |
| **Мониторингом** | Мониторинг наблюдает за живой системой. Performance testing — создаёт нагрузку, чтобы проверить пределы |

#### Ключевая идея:

> Performance testing — это не просто «запустить JMeter и посмотреть на графики». Это процесс выявления **узких мест (bottlenecks)** до того, как они увидят реальные пользователи.

### Ключевые метрики (Response Time, TPS, Error Rate)  <a id="ключевые-метрики"></a>    

Помимо метрик приложения, критически важно отслеживать состояние серверов:

- **CPU Utilization** — загрузка процессора. Пороговое значение: >75–80% в течение длительного времени указывает на нехватку вычислительных мощностей. Анализируйте breakdown: user (приложение), system (ядро), iowait (ожидание диска).

- **Memory Utilization** — использование оперативной памяти. Ключевая метрика для обнаружения утечек памяти (memory leak) — если потребление растет на протяжении теста и не возвращается к исходному уровню после его завершения.

- **Disk I/O** — активность дисковой подсистемы:
  - *IOPS* — количество операций чтения/записи в секунду
  - *Disk Latency (await)* — время ожидания выполнения операции. Значения >10–20 мс для SSD или >50 мс для HDD — тревожный сигнал
  - *Disk Utilization* — процент времени, когда диск занят

- **Network I/O** — входящий и исходящий сетевой трафик. Важно отслеживать, не упирается ли нагрузка в пропускную способность сетевой карты или канала связи.

- **Load Average** (Linux) — средняя нагрузка на систему за 1, 5, 15 минут. Если значение превышает количество логических ядер CPU, система перегружена.

- **Garbage Collection** (для JVM, .NET) — частота и длительность сборок мусора. Длительные GC Pauses напрямую влияют на P99 времени ответа.

## 🚀 Проведение теста   <a id="проведение-теста"></a>   

### Анализ результатов и отчет <a id="анализ-результатов"></a>  

Ключевой навык performance-инженера — связывать метрики приложения с метриками ресурсов. Недостаточно сказать "RT вырос", нужно понять *почему*.

**Таблица симптомов и вероятных причин:**

| Симптом | Метрики для анализа | Вероятная причина | 
|---------|---------------------|-------------------|
| RT растет, TPS падает | CPU → 90-100%, Load Average > ядер | CPU-bound, нехватка процессора |
| RT растет, TPS падает | Disk Latency ↑, IOPS на пределе | I/O-bound, дисковая подсистема — bottleneck |
| RT растет, Memory ↑ и не снижается | Memory Utilization растет постоянно | Утечка памяти (memory leak) |
| P99 резко уходит вверх, средний RT в норме | GC Pauses длительные, Context Switches высокие | Проблемы с синхронизацией или GC |
| Error Rate растет после пика нагрузки | Connection Pool (активные соединения = max) | Исчерпан пул соединений с БД |
| RT периодически "взлетает" | CPU iowait высокий, Disk Latency пиками | Периодические операции I/O (логирование, бэкапы) |

**Алгоритм анализа:**
1. **Смотрим тренды** — как меняются метрики во времени, а не только средние значения  
2. **Ищем корреляцию** — совпадает ли падение TPS с ростом CPU? Рост RT с пиком Disk Latency?
3. **Проверяем пороги** — превышены ли SLO по метрикам ресурсов?
4. **Идем вглубь** — если CPU высокий, смотрим на уровне процессов: какой процесс/поток грузит?

## 📖 Глоссарий и термины  <a id="глоссарий"></a>      

### Основные акронимы и сокращения <a id="акронимы"></a>     

#### 🔹 Основные метрики (дополненные)
- **TPS (Transactions Per Second)** — количество транзакций в секунду. Ключевая метрика пропускной способности.
- **RPS (Requests Per Second)** — количество запросов в секунду. Часто используется как синоним TPS, но может быть более точным для протоколов без понятия «транзакция».
- **RT (Response Time)** — время ответа системы (от отправки запроса до получения полного ответа).
- **P95/P99 (Percentile)** — 95-й/99-й процентиль времени ответа. Означает, что 95% или 99% запросов обработались быстрее этого значения. Критически важны для оценки удовлетворенности пользователей.
- **Error Rate** — процент ошибок от общего числа запросов (например, HTTP 500, таймауты).
- **Throughput** — пропускная способность. Общее количество данных, переданных за единицу времени (например, МБ/с) или количество запросов (RPS).

#### 🔹 Виды тестирования   
- **NFR (Non-Functional Requirements)** — нефункциональные требования. Описывают, *как* система должна работать (скорость, надежность, безопасность), а не *что* она должна делать.
- **SLA (Service Level Agreement)** — соглашение об уровне обслуживания. Юридический или бизнес-контракт между поставщиком и клиентом, гарантирующий определенный уровень сервиса.
- **SLO (Service Level Objective)** — целевой показатель уровня обслуживания. Конкретная метрика и её целевое значение, к которому стремится команда (например, "время ответа < 200 мс для 95% запросов").
- **SLI (Service Level Indicator)** — индикатор уровня обслуживания. Фактически измеряемая метрика (например, фактическое время ответа, процент ошибок).

#### 🔹 Архитектурные термины
- **Bottleneck** — узкое место в системе. Компонент, ограничивающий общую производительность (например, медленный диск, перегруженный CPU, неоптимальный запрос к БД).
- **Latency** — задержка передачи данных. Время, затраченное на передачу данных от отправителя к получателю. Часто включается в общее время ответа.
- **Concurrency** — количество одновременных пользователей, выполняющих операции в системе в один и тот же момент времени.
- **Scalability** — масштабируемость системы. Способность системы увеличивать производительность при добавлении ресурсов (вертикальное или горизонтальное масштабирование).

#### 🔹 Специфические термины (из отчетов JMeter и др.)   
- **Loop / Loop Count** — количество повторений выполнения сценария одним виртуальным пользователем в тесте.
- **Duration** — общая длительность теста. Время, в течение которого генерируется нагрузка.
- **Threads / Users** — количество виртуальных пользователей, одновременно выполняющих сценарий. Синоним *Concurrency* в контексте нагрузочного инструмента.
- **Ramp-up Period** — время, за которое нагрузка достигает заданного количества потоков (пользователей). Используется для плавного увеличения нагрузки.
- **Samples** — общее количество отправленных запросов за тест.
- **Average (Avg)** — среднее арифметическое времени ответа. Не самая надежная метрика, так как сильно зависит от выбросов.
- **Median (Med)** — 50-й процентиль времени ответа. Половина запросов обработалась быстрее этого значения.
- **Min / Max** — минимальное и максимальное время ответа, зафиксированное за тест.
- **Std. Dev. (Standard Deviation)** — стандартное отклонение. Показывает, насколько сильно значения времени ответа разбросаны относительно среднего. Высокое значение говорит о нестабильности системы.
- **KB/sec** — пропускная способность в килобайтах в секунду.
- **Connect Time** — время установки соединения (например, TCP handshake).
- **Latency** (в контексте JMeter) — время от отправки запроса до получения первого байта ответа.
