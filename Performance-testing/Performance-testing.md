## 🎯 Основы <a id="основы"></a>  

### Что такое Performance Testing <a id="что-такое-performance-testing"></a>

**Performance Testing (нагрузочное тестирование производительности)** — это вид нефункционального тестирования, целью которого является оценка поведения системы под нагрузкой: скорости, отзывчивости, стабильности и потребления ресурсов.

Простыми словами: performance testing отвечает на вопрос **«Как быстро и надёжно работает система, когда ею пользуются реальные пользователи?»**

#### Что проверяет Performance Testing:

| Аспект | Что оценивается |
|--------|----------------|
| **Скорость** | Время ответа (response time) на запросы пользователя | 
| **Пропускная способность** | Сколько операций/запросов система обрабатывает в секунду (TPS/RPS) |
| **Стабильность** | Не падает ли производительность со временем |
| **Масштабируемость** | Как растёт производительность при добавлении ресурсов |
| **Надёжность под нагрузкой** | Не появляются ли ошибки при высоком количестве пользователей |

#### Пример из жизни:  

> Вы разрабатываете интернет-магазин. Без нагрузки страница открывается за 0.3 секунды. Но в «чёрную пятницу» одновременно заходят 5000 пользователей. Performance testing покажет: упадёт ли время ответа до 5 секунд, появятся ли ошибки 500, или система выдержит достойно.

#### Чем performance testing НЕ является: 

| Не путать с | Разница |
|-------------|---------|
| **Функциональным тестированием** | Там проверяют «работает ли кнопка?». Здесь — «как *быстро* она срабатывает под нагрузкой?» |
| **Unit-тестированием** | Unit-тесты проверяют методы изолированно. Performance test — систему в сборе |
| **Мониторингом** | Мониторинг наблюдает за живой системой. Performance testing — создаёт нагрузку, чтобы проверить пределы |

#### Ключевая идея:

> Performance testing — это не просто «запустить JMeter и посмотреть на графики». Это процесс выявления **узких мест (bottlenecks)** до того, как они увидят реальные пользователи.

### Цели и когда проводить <a id="цели-и-когда"></a> 

#### 🎯 Цели Performance Testing

| Цель | Что даёт | Пример |
|------|----------|--------|
| **Определить предельную ёмкость системы** | Узнать, сколько пользователей/запросов выдерживает система до падения производительности | «Система держит 2000 concurrent users, после 2200 — RT падает ниже допустимого» |
| **Выявить узкие места (bottlenecks)** | Найти компонент, который ограничивает производительность (CPU, память, диск, сеть, БД, внешние API) | «БД — bottleneck: запрос выполняется 2 секунды из-за отсутствия индекса» |
| **Проверить соответствие NFR / SLO** | Убедиться, что система удовлетворяет бизнес-требованиям к производительности | «P99 response time < 300 мс при 1000 пользователях» |
| **Оценить масштабируемость** | Понять, как добавить ресурсы (вертикально/горизонтально) влияет на производительность | «Удвоение серверов дало прирост TPS только на 30% — проблема в синхронизации» |
| **Обнаружить проблемы стабильности** | Найти memory leak, утечки соединений, деградацию со временем | «Память растёт на 100 МБ/час, через 48 часов — OutOfMemoryError» |
| **Подготовиться к пиковым нагрузкам** | Убедиться, что система выдержит распродажу, праздники, атаки ботов | «Чёрная пятница: система готова к 5000 RPS» |

#### 📅 Когда проводить Performance Testing

**Общее правило:** *Чем раньше, тем лучше, но не раньше, чем есть смысл.*

| Этап разработки | Что тестировать | Почему важно |
|-----------------|----------------|--------------|
| **После написания критического API** | Базовые тесты одного эндпоинта | Не накопить долги по производительности |
| **Перед релизом новой фичи** | Regression — не упала ли производительность | Новая фича не должна ломать старую производительность |
| **При изменении инфраструктуры** | Baseline — что было / что стало | Переезд на новые серверы, смена БД, добавление кэша |
| **Перед ожидаемой пиковой нагрузкой** | Load / Stress тесты под планируемую нагрузку | Чёрная пятница, начало учебного года, распродажа |
| **При подозрении на проблемы** | Целевой тест на гипотезу | «Кажется, стало тормозить после обновления библиотеки» |
| **Регулярно (nightly / weekly)** | Автоматизированные performance-тесты в CI/CD | Отслеживать тренды и ловить регрессии до продуктива |
| **После инцидента в production** | Воспроизводящий тест на staging | «В пятницу было падение — давай воспроизведём и починим» |

#### ❌ Когда НЕ стоит проводить

| Ситуация | Почему |
|----------|--------|
| **Нет стабильного окружения** | Результаты будут несопоставимы и бесполезны | 
| **Нет утверждённых NFR / SLO** | Не с чем сравнивать — «хорошо» или «плохо»? |
| **Система в активной разработке и меняется каждый день** | Тесты устареют быстрее, чем их прогонят |
| **Нет базовых функциональных тестов** | Бессмысленно тестировать производительность того, что не работает |

#### 🔄 Когда в жизненном цикле проекта?

```
Идея → Проектирование → Разработка → Тестирование → Релиз → Эксплуатация
         ↑                ↑              ↑          ↑          ↑
    Закладка NFR     Baseline      Load test   Финальный   Регулярный
                    тесты на API    регрессии   прогон      мониторинг
```

**Ключевая мысль:** Performance testing — это не разовое событие перед релизом. Это **непрерывный процесс**, который сопровождает систему на протяжении всей её жизни.

### Нефункциональные требования (NFR) <a id="nfr"></a>  

#### 📌 Что такое NFR?

**Нефункциональные требования (Non-Functional Requirements)** — это требования, которые описывают **как** должна работать система, в отличие от функциональных требований, которые описывают **что** система должна делать.

| Функциональные требования (что) | Нефункциональные требования (как) |
|--------------------------------|-----------------------------------|
| «Пользователь может войти в систему» | «Вход должен занимать не более 1 секунды» |
| «Система формирует отчёт по продажам» | «Отчёт за месяц формируется менее чем за 5 секунд при 1000 отделах» |
| «Можно добавить товар в корзину» | «Система выдерживает 5000 одновременных пользователей в час пик» |

#### 🧱 Классификация NFR для производительности

В контексте performance testing наиболее важны следующие категории NFR:

| Категория | Что описывает | Пример требования |
|-----------|---------------|-------------------|
| **Производительность (Performance)** | Скорость обработки запросов | «Время ответа API < 200 мс для 95% запросов» |
| **Пропускная способность (Throughput)** | Количество операций в единицу времени | «Система обрабатывает минимум 1000 TPS» |
| **Ёмкость (Capacity)** | Максимальная поддерживаемая нагрузка | «Система выдерживает 10 000 concurrent пользователей» |
| **Масштабируемость (Scalability)** | Способность расти с добавлением ресурсов | «Увеличение серверов в 2 раза даёт рост TPS не менее чем в 1.8 раза» |
| **Стабильность (Stability)** | Отсутствие деградации со временем | «Система работает 72 часа под нагрузкой без падения производительности» |
| **Надёжность (Reliability)** | Отсутствие ошибок под нагрузкой | «Error Rate < 0.01% при любой нагрузке до 5000 пользователей» |
| **Восстанавливаемость (Recoverability)** | Скорость восстановления после перегрузки | «После снятия перегрузки система возвращается к нормальному RT за 30 секунд» |

#### 📝 Как правильно формулировать NFR

**Плохой NFR** ❌:
> «Система должна работать быстро»

**Хороший NFR** ✅:
> «API получения корзины пользователя должен иметь P95 response time < 300 мс при нагрузке 1000 RPS на стандартной конфигурации сервера (4 vCPU, 16 GB RAM)»

**Формула хорошего NFR:**

```
[Компонент/функция] + [Метрика] + [Пороговое значение] + [Условия нагрузки] + [Окружение]
```

**Пример по формуле:**
| Компонент | Метрика | Порог | Условия нагрузки | Окружение |
|-----------|---------|-------|------------------|-----------|
| API поиска | P99 RT | < 500 мс | 200 concurrent пользователей | Стандартный pod |

#### 🚨 Почему NFR критически важны для performance testing

| Без NFR | С NFR |
|---------|-------|
| Непонятно, прошёл тест или провалился | Есть чёткий критерий PASS/FAIL |
| «Стало вроде бы медленнее» | «P99 вырос с 200 мс до 450 мс — SLO нарушен» |
| Тестирование ради тестирования | Тестирование для проверки конкретных бизнес-целей |
| Нельзя автоматизировать принятие решения | Можно сделать CI/CD gate: если NFR нарушены — релиз блокируется |

#### 📋 Примеры NFR для разных систем

| Тип системы | Пример NFR |
|-------------|-------------|
| **Интернет-магазин** | «Главная страница загружается за < 1.5 секунды при 5000 пользователях в Чёрную пятницу» |
| **Платёжный шлюз** | «Обработка платежа занимает < 800 мс для 99.9% транзакций, error rate < 0.001%» |
| **CRM для отдела из 200 сотрудников** | «Открытие карточки клиента — < 500 мс, поиск по всей БД — < 3 секунд» |
| **Мобильное приложение** | «Экран ленты новостей отображается за < 1 секунду при слабом сигнале 3G» |
| **Аналитический отчёт** | «Годовой отчёт формируется за < 10 секунд для базы данных 10 млн записей» |

#### 🔗 Связь NFR с SLA/SLO/SLI

```
NFR (бизнес-требование) → SLO (целевой показатель) → SLI (фактическая метрика)
```

| Уровень | Пример |
|---------|--------|
| **NFR** | «Система должна быть достаточно быстрой для комфортной работы пользователей» |
| **SLO** | «P95 response time < 300 мс для API /get_cart» |
| **SLI** | «За последний час фактический P95 = 245 мс» |

**Важно:** NFR — это то, что вы закладываете на этапе проектирования. SLO — как вы измеряете их соблюдение. А performance testing — как вы проверяете это ДО того, как система попадёт в production.

#### 🛠️ Как собирать NFR для нового проекта

1. **Спросите бизнес:** «Сколько пользователей ожидается? Какие пики? Какая допустимая задержка?»
2. **Изучите конкурентов:** «Как быстро работает похожий сервис у лидера рынка?»
3. **Проанализируйте текущие метрики:** «А что у нас сейчас?»
4. **Учтите бюджет:** «Сколько мы готовы потратить на железо/облако для достижения этих целей?»
5. **Зафиксируйте документально:** NFR должны быть частью технической документации и приниматься всеми сторонами.

### SLA / SLO / SLI и Performance Requirements  <a id="sla-slo"></a>

**SLA (Service Level Agreement)** — юридическое или договорное соглашение между поставщиком услуги и клиентом. Фиксирует гарантированный уровень сервиса и предусматривает ответственность (например, штрафы, компенсации) за его нарушение.

**SLO (Service Level Objective)** — целевое значение конкретной метрики, которое команда обязуется соблюдать. Более детальная и техническая характеристика, чем SLA. Например: «P99 время ответа API /get_cart должно быть не более 300 мс за месяц».

**SLI (Service Level Indicator)** — фактически измеряемый показатель уровня сервиса. Например: «За последний час фактический P95 составил 245 мс, процент успешных запросов — 99.95%».

#### Иерархия и взаимосвязь

```
Performance Requirements (NFR) → SLO → SLA
         ↓                        ↓       ↓
    «как должно быть»        «цель»   «договор»
```

Пример для интернет-магазина:

| Уровень | Пример |
|---------|--------|
| **NFR / Performance Requirement** | «Система должна быть быстрой и надежной при пиковых нагрузках» |
| **SLO** | «P95 response time < 300 мс, Error Rate < 0.1% при 5000 пользователях» |
| **SLI** | «P95 = 267 мс, Error Rate = 0.03% за последние 24 часа» |
| **SLA** | «Гарантируем доступность 99.9%, при нарушении — компенсация 10% месячного платежа» |

#### Как это применяется в Performance Testing  

1. **Performance Requirements** — то, что вы тестируете. Должны быть определены до теста. 
2. **SLO** — критерий прохождения теста (PASS/FAIL). Если SLO нарушен — тест не пройден.
3. **SLI** — то, что вы фактически измеряете в ходе теста и сравниваете с SLO.
4. **SLA** — то, что защищает бизнес в production. Performance testing помогает убедиться, что SLA не будут нарушены после релиза.

#### Типичные SLO для разных систем  

| Тип системы | Примеры SLO |
|-------------|-------------|
| Веб-API | P99 RT < 200 мс, Error Rate < 0.05% при 1000 RPS |
| Платежный шлюз | P99 RT < 800 мс, Error Rate < 0.001%, TPS > 500 |
| Мобильное приложение | Время загрузки экрана < 1.5 с на слабом 3G |
| Отчетная система | Генерация отчета за месяц < 5 с для 1 млн записей |

#### Важные отличия

| Понятие | Уровень | Ответственность | Юридическая сила | Гибкость |
|---------|---------|----------------|------------------|----------|
| **SLA** | Бизнес / договор | Продакт, юристы | Есть | Низкая |
| **SLO** | Техническая команда | DevOps, разработка | Нет | Высокая |
| **SLI** | Измерение | Инженеры | Нет | Очень высокая |

#### Ошибки при работе с SLO/SLA

1. **Ставить SLO, которые нельзя измерить** — «максимально быстро» вместо «P99 < 300 мс»
2. **Путать SLO и SLA** — использовать в договоре то, что должно быть внутренним ориентиром
3. **Измерять средние вместо процентилей** — среднее значение маскирует проблемы (10 запросов по 1 мс и один за 10 секунд дадут среднее ~1 секунду)
4. **Не обновлять SLO при изменении системы** — старые цели становятся нереалистичными или слишком легкими
5. **Ставить слишком много SLO** — лучше 3-5 ключевых метрик, чем 20, за которыми невозможно следить

### Ключевые метрики (Response Time, TPS, Error Rate)  <a id="ключевые-метрики"></a>    

Помимо метрик приложения, критически важно отслеживать состояние серверов:

- **CPU Utilization** — загрузка процессора. Пороговое значение: >75–80% в течение длительного времени указывает на нехватку вычислительных мощностей. Анализируйте breakdown: user (приложение), system (ядро), iowait (ожидание диска).

- **Memory Utilization** — использование оперативной памяти. Ключевая метрика для обнаружения утечек памяти (memory leak) — если потребление растет на протяжении теста и не возвращается к исходному уровню после его завершения.

- **Disk I/O** — активность дисковой подсистемы: 
  - *IOPS* — количество операций чтения/записи в секунду
  - *Disk Latency (await)* — время ожидания выполнения операции. Значения >10–20 мс для SSD или >50 мс для HDD — тревожный сигнал
  - *Disk Utilization* — процент времени, когда диск занят

- **Network I/O** — входящий и исходящий сетевой трафик. Важно отслеживать, не упирается ли нагрузка в пропускную способность сетевой карты или канала связи.

- **Load Average** (Linux) — средняя нагрузка на систему за 1, 5, 15 минут. Если значение превышает количество логических ядер CPU, система перегружена.

- **Garbage Collection** (для JVM, .NET) — частота и длительность сборок мусора. Длительные GC Pauses напрямую влияют на P99 времени ответа.

### Роль Performance-инженера <a id="роль-performance-инженера"></a>

**Performance-инженер** — специалист, отвечающий за обеспечение требуемого уровня производительности системы на всех этапах её жизненного цикла: от анализа требований до эксплуатации в production. В отличие от тестировщика, который проверяет функциональность, performance-инженер фокусируется на скорости, стабильности, пропускной способности и эффективности использования ресурсов.

#### Основные зоны ответственности

| Зона | Что делает |
|------|-------------|
| **Анализ требований** | Собирает и уточняет NFR, помогает формулировать SLO, переводит бизнес-метрики в технические |
| **Проектирование тестов** | Разрабатывает профили нагрузки, сценарии использования, стратегию тестирования |
| **Разработка скриптов** | Пишет и отлаживает нагрузочные скрипты в JMeter, Locust, Gatling и др. |
| **Подготовка окружения** | Настраивает тестовый стенд, генераторы нагрузки, инструменты мониторинга |
| **Запуск тестов** | Проводит Baseline, Load, Stress, Soak тесты, управляет нагрузкой |
| **Мониторинг** | Отслеживает метрики приложения и инфраструктуры (CPU, память, диск, сеть, GC) |
| **Анализ результатов** | Выявляет узкие места (bottlenecks), коррелирует метрики, определяет причины деградации |
| **Оптимизация** | Даёт рекомендации разработчикам, DBA, DevOps по улучшению производительности |
| **Автоматизация** | Встраивает performance-тесты в CI/CD, настраивает ночные прогоны |
| **Расследование инцидентов** | Помогает воспроизводить и анализировать проблемы производительности в production |

#### Навыки Performance-инженера <a id="навыки-performance-инженера"></a>  

**Технические навыки:**

| Категория | Конкретные навыки |
|-----------|-------------------|
| **Инструменты нагрузочного тестирования** | JMeter (основной), Locust, Gatling, k6, Yandex.Tank |
| **Языки программирования** | Java (для JMeter), Python (для Locust), JavaScript (для k6), скрипты на Bash |
| **Мониторинг и observability** | Grafana, Prometheus, Zabbix, APM (New Relic, Datadog, Dynatrace, SkyWalking) |
| **Работа с базами данных** | SQL (поиск медленных запросов, анализ планов), понимание индексов |
| **Сети и протоколы** | HTTP/HTTPS, REST, gRPC, WebSocket, TCP/IP, понимание latency, bandwidth |
| **Инфраструктура** | Linux (системные метрики, настройка ядра), Docker, Kubernetes, облака (AWS/GCP/Azure) |
| **Анализ данных** | Excel, Jupyter, умение строить графики, работать с логами и метриками |
| **CI/CD** | Jenkins, GitLab CI, GitHub Actions (запуск тестов по расписанию/после деплоя) |

**Soft skills:**

| Навык | Зачем нужен |
|-------|-------------|
| **Системное мышление** | Видеть систему целиком, а не отдельные компоненты |
| **Умение задавать вопросы** | Выяснять реальные NFR, когда их нет или они размыты |
| **Коммуникация** | Доносить результаты до разработчиков, менеджеров, заказчика |
| **Скептицизм** | Не верить «у нас всё работает быстро», проверять цифрами |
| **Терпение и внимательность** | Тесты могут идти часами, анализ требует деталей |

#### Чем performance-инженер отличается от смежных ролей

| Роль | Фокус | Основной вопрос |
|------|-------|-----------------|
| **Функциональный тестировщик** | Правильность работы | «Работает ли кнопка?» |
| **DevOps / SRE** | Стабильность и доступность в production | «Не упал ли сервер?» |
| **Разработчик** | Корректность и читаемость кода | «Правильно ли реализована логика?» |
| **Performance-инженер** | Скорость и эффективность под нагрузкой | «Не тормозит ли под 1000 пользователей?» |

#### Типичный рабочий процесс performance-инженера

```
1. Получение задачи
   ↓
2. Уточнение NFR и SLO (если нет — помочь сформулировать)
   ↓
3. Анализ архитектуры и типовых сценариев использования
   ↓
4. Проектирование теста: профиль нагрузки, метрики, окружение
   ↓
5. Разработка скриптов и подготовка данных
   ↓
6. Запуск Baseline теста (лёгкая нагрузка для замера "дна")
   ↓
7. Запуск основного теста (Load / Stress / Soak)
   ↓
8. Мониторинг и сбор метрик (приложение + инфраструктура)
   ↓
9. Анализ: поиск корреляций, узких мест, нарушений SLO
   ↓
10. Подготовка отчёта: выводы, графики, рекомендации
    ↓
11. Передача результатов команде, обсуждение оптимизаций
```

#### Типичные проблемы, которые находит performance-инженер 

| Проблема | Как проявляется |
|----------|-----------------|
| **Отсутствие индексов в БД** | RT резко растёт при увеличении объёма данных |
| **Утечка памяти (memory leak)** | Memory Utilization растёт на протяжении Soak-теста |
| **Неоптимальные алгоритмы** | CPU уходит в 100%, TPS падает |
| **Блокировки и deadlock'и** | Время ответа периодически "взлетает", ошибки таймаутов |
| **Неправильные пулы соединений** | При пике ошибки "Connection pool exhausted" |
| **Ограничения сети** | Throughput упирается в лимиты, растёт latency |
| **GC проблемы (для JVM/.NET)** | P99 сильно выше среднего из-за долгих пауз GC |

#### Ключевая идея

> Performance-инженер — это не «человек, который запускает JMeter». Это инженер, который понимает, как работает система под капотом, умеет находить узкие места на всех уровнях (код, БД, сеть, инфраструктура) и давать конкретные, проверяемые рекомендации по улучшению. Его главная метрика успеха — количество проблем производительности, найденных на staging до того, как они увидят реальные пользователи.

## 🚀 Проведение теста   <a id="проведение-теста"></a>   

### Анализ результатов и отчет <a id="анализ-результатов"></a>  

Ключевой навык performance-инженера — связывать метрики приложения с метриками ресурсов. Недостаточно сказать "RT вырос", нужно понять *почему*.

**Таблица симптомов и вероятных причин:**

| Симптом | Метрики для анализа | Вероятная причина | 
|---------|---------------------|-------------------|
| RT растет, TPS падает | CPU → 90-100%, Load Average > ядер | CPU-bound, нехватка процессора |
| RT растет, TPS падает | Disk Latency ↑, IOPS на пределе | I/O-bound, дисковая подсистема — bottleneck |
| RT растет, Memory ↑ и не снижается | Memory Utilization растет постоянно | Утечка памяти (memory leak) |
| P99 резко уходит вверх, средний RT в норме | GC Pauses длительные, Context Switches высокие | Проблемы с синхронизацией или GC |
| Error Rate растет после пика нагрузки | Connection Pool (активные соединения = max) | Исчерпан пул соединений с БД |
| RT периодически "взлетает" | CPU iowait высокий, Disk Latency пиками | Периодические операции I/O (логирование, бэкапы) |

**Алгоритм анализа:**
1. **Смотрим тренды** — как меняются метрики во времени, а не только средние значения  
2. **Ищем корреляцию** — совпадает ли падение TPS с ростом CPU? Рост RT с пиком Disk Latency?
3. **Проверяем пороги** — превышены ли SLO по метрикам ресурсов?
4. **Идем вглубь** — если CPU высокий, смотрим на уровне процессов: какой процесс/поток грузит?

## 📖 Глоссарий и термины  <a id="глоссарий"></a>      

### Основные акронимы и сокращения <a id="акронимы"></a>     

#### 🔹 Основные метрики (дополненные)
- **TPS (Transactions Per Second)** — количество транзакций в секунду. Ключевая метрика пропускной способности.
- **RPS (Requests Per Second)** — количество запросов в секунду. Часто используется как синоним TPS, но может быть более точным для протоколов без понятия «транзакция».
- **RT (Response Time)** — время ответа системы (от отправки запроса до получения полного ответа).
- **P95/P99 (Percentile)** — 95-й/99-й процентиль времени ответа. Означает, что 95% или 99% запросов обработались быстрее этого значения. Критически важны для оценки удовлетворенности пользователей.
- **Error Rate** — процент ошибок от общего числа запросов (например, HTTP 500, таймауты).
- **Throughput** — пропускная способность. Общее количество данных, переданных за единицу времени (например, МБ/с) или количество запросов (RPS).

#### 🔹 Виды тестирования   
- **NFR (Non-Functional Requirements)** — нефункциональные требования. Описывают, *как* система должна работать (скорость, надежность, безопасность), а не *что* она должна делать.
- **SLA (Service Level Agreement)** — соглашение об уровне обслуживания. Юридический или бизнес-контракт между поставщиком и клиентом, гарантирующий определенный уровень сервиса.
- **SLO (Service Level Objective)** — целевой показатель уровня обслуживания. Конкретная метрика и её целевое значение, к которому стремится команда (например, "время ответа < 200 мс для 95% запросов").
- **SLI (Service Level Indicator)** — индикатор уровня обслуживания. Фактически измеряемая метрика (например, фактическое время ответа, процент ошибок).

#### 🔹 Архитектурные термины
- **Bottleneck** — узкое место в системе. Компонент, ограничивающий общую производительность (например, медленный диск, перегруженный CPU, неоптимальный запрос к БД).
- **Latency** — задержка передачи данных. Время, затраченное на передачу данных от отправителя к получателю. Часто включается в общее время ответа.
- **Concurrency** — количество одновременных пользователей, выполняющих операции в системе в один и тот же момент времени.
- **Scalability** — масштабируемость системы. Способность системы увеличивать производительность при добавлении ресурсов (вертикальное или горизонтальное масштабирование).

#### 🔹 Специфические термины (из отчетов JMeter и др.)   
- **Loop / Loop Count** — количество повторений выполнения сценария одним виртуальным пользователем в тесте.
- **Duration** — общая длительность теста. Время, в течение которого генерируется нагрузка.
- **Threads / Users** — количество виртуальных пользователей, одновременно выполняющих сценарий. Синоним *Concurrency* в контексте нагрузочного инструмента.
- **Ramp-up Period** — время, за которое нагрузка достигает заданного количества потоков (пользователей). Используется для плавного увеличения нагрузки.
- **Samples** — общее количество отправленных запросов за тест.
- **Average (Avg)** — среднее арифметическое времени ответа. Не самая надежная метрика, так как сильно зависит от выбросов.
- **Median (Med)** — 50-й процентиль времени ответа. Половина запросов обработалась быстрее этого значения.
- **Min / Max** — минимальное и максимальное время ответа, зафиксированное за тест.
- **Std. Dev. (Standard Deviation)** — стандартное отклонение. Показывает, насколько сильно значения времени ответа разбросаны относительно среднего. Высокое значение говорит о нестабильности системы.
- **KB/sec** — пропускная способность в килобайтах в секунду.
- **Connect Time** — время установки соединения (например, TCP handshake).
- **Latency** (в контексте JMeter) — время от отправки запроса до получения первого байта ответа.
