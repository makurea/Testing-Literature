### Коллекции: list, tuple, set, dict <a id="коллекции-list-tuple-set-dict"></a>

**Сравнительная таблица коллекций**

| Тип     | Упорядоченность    | Изменяемость | Уникальность элементов | Синтаксис      | Пример             |
| ------- | ------------------ | ------------ | ---------------------- | -------------- | ------------------ |
| `list`  | Да                 | Да           | Нет                    | `[...]`        | `[1, 2, 3]`        |
| `tuple` | Да                 | Нет          | Нет                    | `(...)`        | `(1, 2, 3)`        |
| `set`   | Нет                | Да           | Да                     | `{...}`        | `{1, 2, 3}`        |
| `dict`  | Да (с Python 3.7+) | Да           | Ключи уникальны        | `{key: value}` | `{"a": 1, "b": 2}` |

####  1. `list` — список

**Список** — самая популярная коллекция в Python.

**Особенности:**

* хранит элементы в **определённом порядке**,
* **изменяемый** — можно добавлять, удалять и менять элементы,
* допускает элементы **разных типов**,
* элементы могут повторяться.

**Основные операции:**

```python
numbers = [1, 2, 3]

numbers.append(4)      # добавление
numbers[0] = 10        # изменение
numbers.remove(2)      # удаление по значению
numbers.pop()          # удаление последнего
```

####  2. `tuple` — кортеж

**Кортеж** похож на список, но **неизменяемый**.

**Особенности:**

* хранит данные в **фиксированном порядке**,
* после создания его **нельзя изменить**,
* работает быстрее списка,
* используется, когда данные должны быть **зафиксированы**.

**Пример:**

```python
point = (10, 20)
print(point[0])   # 10
```

####  3. `set` — множество

**Множество** — коллекция уникальных элементов.

**Особенности:**

* элементы **не повторяются**,
* **неупорядоченная** структура (порядок может быть случайным),
* можно выполнять математические операции: объединение, пересечение.

**Пример:**

```python
unique_nums = {1, 2, 3, 3, 2}
print(unique_nums)  # {1, 2, 3}

a = {1, 2, 3}
b = {3, 4}
print(a | b)  # объединение -> {1, 2, 3, 4}
print(a & b)  # пересечение -> {3}
```

####  4. `dict` — словарь

**Словарь** — коллекция пар «ключ → значение».

**Особенности:**

* данные хранятся в формате: **ключ : значение**,
* ключи **должны быть уникальными**,
* ключом может быть строка, число, кортеж,
* быстрый доступ к данным по ключу.

#### Пример:

```python
user = {
    "name": "Tom",
    "age": 22
}

print(user["name"])      # Tom
user["age"] = 22         # изменение
user["city"] = "London"   # добавление
```

**Итоги**

* `list` — универсальный изменяемый список.
* `tuple` — неизменяемые данные, быстро и безопасно.
* `set` — уникальные значения, операции с множествами.
* `dict` — ключ-значение, идеален для структурированных данных.

### Мутабельные и иммутабельные типы <a id="мутабельные-и-иммутабельные-тип"></a>

В Python все типы данных делятся на две большие категории: **мутабельные** (изменяемые) и **иммутабельные** (неизменяемые).
Разница в том, можно ли изменить объект *после его создания*.

**Сравнительная таблица**

| Тип           | Примеры                                                      | Можно изменить? | Что именно меняется                  |
| ------------- | ------------------------------------------------------------ | --------------- | ------------------------------------ |
| Иммутабельные | `int`, `float`, `bool`, `str`, `tuple`, `frozenset`, `bytes` | ❌ Нет           | Любое изменение создаёт новый объект |
| Мутабельные   | `list`, `set`, `dict`, `bytearray`                           | ✔ Да            | Объект меняется прямо в памяти       |

**Иммутабельные типы**

Объект нельзя изменить. Любая «модификация» создаёт новый объект в памяти.

Пример:

```python
a = "hello"
a = a.upper()  # создаётся новая строка
```

В результате переменная `a` теперь ссылается на совершенно новый объект, хотя выглядит так, будто строка просто изменилась.

Иммутабельность делает такие объекты:

* безопасными,
* быстрыми,
* удобными для использования как ключи словаря.

**Мутабельные типы**

Объект можно менять без создания нового.

Пример:

```python
numbers = [1, 2, 3]
numbers.append(4)  # список меняется прямо в памяти
```

Мутабельные типы удобны, если нужно часто менять содержимое структуры данных.

**Почему это важно?**

* При работе с функциями иммутабельные объекты защищают данные от случайного изменения.
* Мутабельные коллекции удобнее для обработки больших объёмов данных.
* Это напрямую влияет на производительность, отладку и поведение кода.

### Тип None и проверка на пустоту <a id="тип-none"></a>

**`None`** — это специальное значение в Python, обозначающее отсутствие данных.
Это не ноль, не пустая строка, не `False` — это отдельный самостоятельный тип: **`NoneType`**.

Такое значение часто используется:

* когда функция ничего не возвращает (`return` без значения),
* как дефолтное значение аргумента,
* чтобы явно указать: «Здесь данных нет».

**Пример использования**  

```python
result = None

if result is None:
    print("Нет данных")
```

**Проверка на None**

Важно: `None` всегда проверяют через **`is`** или **`is not`**, а не через `==`.

```python
data = None

if data is None:
    print("Пусто")

if data is not None:
    print("Есть значение")
```

**None vs пустые значения**

`None` часто путают с «пустыми» объектами: `""`, `[]`, `{}`, `0`, `False`.
Но это **не одно и то же**.

Сравнение:

| Значение | Тип        | «Пустое»? | Логическое значение |
| -------- | ---------- | --------- | ------------------- |
| `None`   | `NoneType` | Да        | `False`             |
| `""`     | `str`      | Да        | `False`             |
| `[]`     | `list`     | Да        | `False`             |
| `{}`     | `dict`     | Да        | `False`             |
| `0`      | `int`      | Да        | `False`             |
| `False`  | `bool`     | —         | `False`             |

**Проверка на пустоту**

Python считает пустыми многие объекты: строку, список, словарь, множество.

```python
if not my_list:
    print("Список пустой")
```

Этот способ универсален: он работает и для `None`, и для пустых коллекций.

**Когда использовать что**

* `None` — когда значения **нет вовсе**.
* Пустые `[]`, `{}`, `""` — когда значение есть, но оно пустое.

### Преобразование типов - Type Casting <a id="преобразование-типов"></a>  

В Python часто требуется менять тип данных: превращать строку в число, число в строку, список в множество и т.д.
Такой процесс называется **type casting** или **преобразованием типов**.

Python позволяет делать конвертацию как **неявно**, так и **явно**.

**Неявное преобразование**

Python самостоятельно меняет тип, когда это безопасно.
Например, при вычислениях с `int` и `float` результат всегда `float`.

```python
a = 5       # int
b = 2.5     # float

result = a + b
print(result)       # 7.5 (float)
```

Здесь число автоматически стало типом `float`.

**Явное преобразование**

Выполняется вручную через функции-конструкторы:

* `int()` — в целое число
* `float()` — в число с точкой
* `str()` — в строку
* `bool()` — в логический тип
* `list()` — список
* `tuple()` — кортеж
* `set()` — множество
* `dict()` — словарь (только у правильных структур)

**Примеры преобразований**

**Строка → число**

```python
a = "42"
num = int(a)   # 42
```

**Число → строка**

```python
n = 10
s = str(n)   # "10"
```

**Список → множество** 

```python
items = [1, 2, 2, 3]
unique = set(items)   # {1, 2, 3}
```

**Кортеж → список**  

```python
t = (10, 20)
lst = list(t)   # [10, 20]
```

**Преобразование в bool**

`bool()` чаще всего используется для проверки «пустоты».
В логическом контексте пустые объекты превращаются в `False`.

| Значение                | bool()  |
| ----------------------- | ------- |
| `0`                     | `False` |
| `""`                    | `False` |
| `[]`                    | `False` |
| `{}`                    | `False` |
| `None`                  | `False` |
| Любое непустое значение | `True`  |

**Ошибки при преобразовании**

Некоторые преобразования не могут быть выполнены:

```python
int("abc")    # Ошибка: нельзя превратить в число
```

А словарь можно создать только из корректных пар:

```python
dict([(1, "a"), (2, "b")])   # OK
dict([1, 2, 3])              # Ошибка
```

**Важно помнить**

* Преобразование типов — это безопасный способ контролировать формат данных.
* Явное преобразование даёт больше контроля.
* Неявное работает только там, где это однозначно.

## 🟠 Операторы и циклы <a id="операторы-и-циклы"></a>

### Арифметические, логические и сравнения <a id="операторы"></a>

#### **Арифметические операторы**

Используются для выполнения математических вычислений:

| Оператор | Описание               | Пример        |
| -------- | ---------------------- | ------------- |
| `+`      | Сложение               | `3 + 2 → 5`   |
| `-`      | Вычитание              | `5 - 1 → 4`   |
| `*`      | Умножение              | `4 * 2 → 8`   |
| `/`      | Деление (всегда float) | `5 / 2 → 2.5` |
| `//`     | Целочисленное деление  | `5 // 2 → 2`  |
| `%`      | Остаток от деления     | `5 % 2 → 1`   |
| `**`     | Возведение в степень   | `2 ** 3 → 8`  |

#### **Операторы сравнения**

Возвращают `True` или `False`.

| Оператор | Значение         | Пример          |
| -------- | ---------------- | --------------- |
| `==`     | Равно            | `3 == 3 → True` |
| `!=`     | Не равно         | `3 != 2 → True` |
| `>`      | Больше           | `5 > 3 → True`  |
| `<`      | Меньше           | `2 < 1 → False` |
| `>=`     | Больше или равно | `4 >= 4 → True` |
| `<=`     | Меньше или равно | `3 <= 5 → True` |

Особенности:

* Сравнение строк идёт по **лексикографическому порядку**: `"b" > "a"` → `True`.
* Объекты разных типов сравнивать напрямую нельзя (кроме некоторых случаев — например, числа).

#### **Логические операторы**

Используются для комбинирования логических выражений:

| Оператор | Значение                       | Пример                   |
| -------- | ------------------------------ | ------------------------ |
| `and`    | Истина, если оба условия True  | `True and False → False` |
| `or`     | Истина, если хотя бы одно True | `True or False → True`   |
| `not`    | Логическое отрицание           | `not True → False`       |

Python использует **ленивые вычисления**:  

* `A and B` — B вычисляется только если A истинно
* `A or B` — B вычисляется только если A ложно

Пример:

```python
def hello():
    print("Вызвано!")  

False and hello()   # функция НЕ вызовется
True or hello()     # функция НЕ вызовется
```

#### Краткие выводы

* Арифметика работает как в математике, но `/` всегда даёт float.
* Операторы сравнения возвращают булевы значения.
* `and`, `or`, `not` управляют логическими выражениями и используют ленивую оценку.

### Условные конструкции if / elif / else <a id="условные-конструкции"></a>

**if**
Код внутри блока if выполняется только если условие истинно (True).

Пример:

```
age = 18
if age >= 18:
    print("Совершеннолетний")
```

**elif**
Используется, когда нужно проверить несколько альтернативных условий. Выполняется только первое истинное условие.

Пример:

```
temp = 15
if temp > 25:
    print("Жарко")
elif temp > 15:
    print("Тепло")
else:
    print("Прохладно")
```

**else**
Срабатывает, если все предыдущие условия оказались ложными. Условие у else не указывается.  

Пример:

```
number = -5
if number > 0:
    print("Положительное")
else:
    print("Неположительное")
```

**Важные особенности** 

1. Отступы обязательны — именно они определяют структуру блоков.
2. Условия можно комбинировать через and, or, not.
3. Тело условия может содержать сколько угодно строк.
4. Внутри условий допускается вложенность.

Пример вложенности:

```
x = 10
if x > 0:
    if x % 2 == 0:
        print("Положительное чётное")
    else:
        print("Положительное нечётное")
```

### Циклы for и while <a id="циклы"></a>

**Цикл for**
Итерируется по последовательности (списку, строке, диапазону, множеству, словарю и т.д.) и выполняет код для каждого элемента.

Примеры:
Итерирование по списку:

```
for item in [1, 2, 3]:
    print(item)
```

Итерирование по строке:  

```
for char in "Hello":
    print(char)
```

Использование range:  

```
for i in range(5):
    print(i)
```

Здесь будут выведены числа от 0 до 4.

Итерирование по словарю:

```
person = {"name": "Nick", "age": 26}
for key, value in person.items():
    print(key, value)
```

**Цикл while**
Выполняется до тех пор, пока условие истинно. Подходит для случаев, когда количество итераций заранее неизвестно.

Пример:

```
count = 0
while count < 3:
    print("Итерация:", count)
    count += 1
```

Остановка цикла вручную:

```
while True:
    cmd = input("Введите команду: ")
    if cmd == "exit":
        break
```

**Особенности циклов**

1. Оба цикла поддерживают break (прерывание) и continue (пропуск итерации).
2. В циклах for можно итерироваться по любым итерируемым объектам.
3. В while важно следить, чтобы условие когда-нибудь стало ложным, иначе будет бесконечный цикл.
4. Циклы могут содержать else-блок, который выполняется, если цикл завершился штатно (без break).

Пример с else:

```
for i in range(3):
    print(i)
else:
    print("Цикл завершён")
```

### Break / Continue / Else в циклах <a id="break-continue-else"></a>  

**break**
Полностью прекращает выполнение цикла. Управление передаётся коду после цикла.

Пример:

```
for num in range(10):
    if num == 5:
        break
    print(num)
```

Цикл остановится при достижении числа 5.

**continue**
Пропускает текущую итерацию и переходит сразу к следующей.

Пример:

```
for num in range(5): 
    if num == 2:
        continue
    print(num)
```

Число 2 не будет напечатано.

**else в циклах**
Блок else выполняется только если цикл завершился естественным образом — то есть без использования break.

Пример с for:

```
for i in range(3):
    print(i)
else:
    print("Цикл завершён без break")
```

Пример с while:

```
x = 0
while x < 3:
    print(x)
    x += 1
else:
    print("Цикл завершился нормально")
```

Пример, где else не выполнится:

```
for i in range(5):
    if i == 3:
        break
else:
    print("Это не выполнится")
```

### Генераторы списков и выражений - Comprehensions <a id="comprehensions"></a>

**Генератор списка (list comprehension)**
Создаёт новый список внутри одной строки.

Шаблон:

```
[выражение for элемент in последовательность if условие]
```

Примеры:
Создать список квадратов:

```
squares = [x * x for x in range(5)]
```

Отфильтровать чётные числа:

```
evens = [x for x in range(10) if x % 2 == 0]
```

Работа со строками:

```
chars = [c.upper() for c in "hello"]
```

Комбинация двух циклов:

```
pairs = [(x, y) for x in range(3) for y in range(2)]
```

**Генератор множества (set comprehension)**
Создаёт множество по аналогичному синтаксису.

```
unique = {x % 3 for x in range(10)}
```

**Генератор словаря (dict comprehension)**  
Удобен для формирования словарей.

```
d = {x: x * 2 for x in range(5)}
```

**Генераторные выражения (generator expressions)**
Похожи на list comprehension, но создают ленивый генератор, который выдаёт элементы по одному, не занимая память как список.

```
gen = (x * 2 for x in range(5)) 
```

Элементы генератора можно проходить только один раз:

```
for value in gen:
    print(value)
```

**Когда использовать comprehensions**

1. Когда логика проста и умещается в одну строку.
2. Когда нужна компактность и читаемость.
3. Когда важно избежать лишнего кода.
4. Generator expressions предпочтительнее, если не требуется хранить результат целиком в памяти.

### Определение функций (def) <a id="определение-функций"></a>

Определение функций позволяет вынести повторяющийся код в отдельные именованные блоки и вызывать их по необходимости. Функции помогают структурировать программу, улучшать читаемость и повторно использовать логику.

**Создание функции с помощью def**
Функция объявляется с ключевым словом def, затем указывается её имя и параметры в скобках. 

Пример простой функции:

```
def greet():
    print("Привет!")
```

Вызов функции:

```
greet()
```

**Функции с аргументами**
Параметры позволяют передавать функции значения для обработки.  

```
def add(a, b):
    return a + b
```

```
result = add(3, 4)
print(result)
```

**Параметры по умолчанию**
Если значение не передано, используется дефолтное. 

```
def greet(name="Гость"):
    print("Привет,", name)
```

**Докстрока (описание функции)**
Помогает документировать назначение функции.

```
def multiply(a, b):
    """Возвращает произведение двух чисел."""
    return a * b
```

**Важные особенности**

1. Тело функции должно иметь отступ.
2. Имя функции должно быть читаемым и отражать её назначение.
3. Функция может ничего не возвращать — тогда она фактически возвращает None.
4. Внутри функции создаётся собственная локальная область видимости.

### Аргументы и параметры (*args, **kwargs) <a id="аргументы"></a>  

Аргументы и параметры позволяют гибко передавать данные в функции. Python поддерживает несколько типов аргументов, включая позиционные, именованные и специальные конструкции *args и **kwargs для работы с переменным числом значений.

**Позиционные аргументы**
Передаются в порядке, соответствующем определению функции.

```  
def add(a, b):
    return a + b

add(3, 5)
```

**Именованные аргументы**
Задаются с указанием имени параметра.

```
def greet(name, age):
    print("Имя:", name, "Возраст:", age)

greet(age=26, name="Николай")
```

**Параметры по умолчанию**  
Если значение не передано — используется дефолт.

```
def greet(name="Гость"):
    print("Привет,", name)
```

**Аргументы *args**
Позволяют передавать любое количество позиционных аргументов. Внутри функции args — это кортеж.

```
def summ(*args):
    return sum(args)

print(summ(1, 2, 3, 4))
```

**Аргументы **kwargs**
Принимают неограниченное количество именованных аргументов. Внутри функции kwargs — это словарь.

```
def info(**kwargs):
    for key, value in kwargs.items():
        print(key, value)

info(name="Nick", age=26)
```

**Комбинирование аргументов**
Порядок в определении функции всегда одинаковый:

параметры → *args → параметры с дефолтом → **kwargs

Пример:

```
def func(a, b, *args, c=10, **kwargs):
    print(a, b, args, c, kwargs)
```

**Распаковка аргументов**
Звёздочка позволяет «распаковывать» коллекции при передаче:  

```
values = [1, 2, 3]
print(*values)
```

```
data = {"name": "Nick", "age": 26}
info(**data)
```

**Итог**

1. *args — переменное число позиционных аргументов.
2. **kwargs — переменное число именованных аргументов.
3. Можно комбинировать разные типы аргументов для гибких и универсальных функций.

### Возврат значений (return) <a id="return"></a>

Оператор return используется для возврата результата из функции. Он завершает выполнение функции и передаёт значение обратно в место вызова.

**Простой возврат значения**
Функция может вернуть любое значение: число, строку, список, объект и т.д.

```
def add(a, b):
    return a + b

result = add(3, 4)
print(result)
```

**Возврат нескольких значений**
Python позволяет возвращать несколько значений одновременно — фактически возвращается кортеж.

```
def get_user():
    return "Nick", 26

name, age = get_user()
```

**Отсутствие return**
Если в функции нет оператора return, она автоматически возвращает None.

```
def greet():
    print("Привет!")

result = greet()
print(result)   # None
```

**return завершает функцию**
Всё, что написано после return, не выполняется.

```
def test():
    return 5
    print("Это не выполнится") 
```

**Условный возврат**
Можно возвращать разные значения в зависимости от условий.

```
def check(n):
    if n > 0:
        return "Положительное"
    return "Неположительное"
```

**Возврат сложных объектов**
Можно возвращать списки, словари, функции, лямбды или даже классы.

```
def factory():
    return lambda x: x * 2

f = factory()
print(f(5))
```

**Итоги**

1. return завершает функцию и отправляет значение обратно.
2. Можно возвращать любое количество значений.
3. Если return отсутствует, возвращается None.

### Анонимные функции (lambda) <a id="лямбда"></a>

Анонимные функции — это компактные однострочные функции, которые создаются с помощью ключевого слова lambda. Они не имеют имени и используются там, где требуется простая функция «на месте».

**Синтаксис lambda**

```
lambda аргументы: выражение
```

Выражение вычисляется и его результат возвращается автоматически.

**Пример простой лямбды**

```
double = lambda x: x * 2
print(double(5))
```

**Лямбда с несколькими аргументами**

```
add = lambda a, b: a + b
print(add(3, 4))
```

**Использование в сортировке**
Лямбда часто применяется как ключ сортировки.

```
items = ["apple", "banana", "kiwi"]
items.sort(key=lambda x: len(x))
```

**Использование в map, filter**

```
nums = [1, 2, 3, 4]

doubled = list(map(lambda x: x * 2, nums))
even = list(filter(lambda x: x % 2 == 0, nums))
```

**Лямбда внутри других функций**  

```
def make_multiplier(n):
    return lambda x: x * n

mul3 = make_multiplier(3)
print(mul3(10))
```

**Особенности**

1. Лямбда должна содержать только одно выражение, без многострочных блоков.
2. Используется для краткости — там, где создавать обычную функцию def избыточно.
3. Полностью эквивалентна обычной функции, за исключением отсутствия имени.

### map / filter / reduce <a id="map-filter-reduce"></a>

Функции для обработки последовательностей в функциональном стиле.

Таблица общая:

| Функция | Что делает                           | Что возвращает    | Когда применять         |
| ------- | ------------------------------------ | ----------------- | ----------------------- |
| map     | применяет функцию к каждому элементу | объект map        | массовое преобразование |
| filter  | оставляет элементы по условию        | объект filter     | фильтрация данных       |
| reduce  | сворачивает в одно значение          | итоговое значение | накопление результата   |

Примеры:

| Пример                                | Результат |
| ------------------------------------- | --------- |
| `map(lambda x: x*2, [1,2,3])`         | `[2,4,6]` |
| `filter(lambda x: x%2==0, [1,2,3,4])` | `[2,4]`   |
| `reduce(lambda x,y: x+y, [1,2,3])`    | `6`       |

Особенности:

| Особенность                 | map | filter | reduce                           |
| --------------------------- | --- | ------ | -------------------------------- |
| ленивый результат           | ✔   | ✔      | ✘                                |
| не изменяет исходный список | ✔   | ✔      | n/a                              |
| нужен импорт                | ✘   | ✘      | ✔ `from functools import reduce` |

Мини-итог:

| map         | filter   | reduce     |
| ----------- | -------- | ---------- |
| преобразует | отбирает | агрегирует |

### Декораторы <a id="декораторы"></a>  

Декораторы — это способ изменить или расширить поведение функции без изменения её кода. По сути, декоратор принимает функцию, оборачивает её в другую и возвращает новую функцию.

**Как работают**

* Функция-декоратор принимает другую функцию как аргумент.
* Внутри создаётся «обёртка» (wrapper), которая выполняет дополнительный код.
* Декоратор возвращает эту обёртку вместо исходной функции.
* Используется через `@имя_декоратора` над функцией.

**Простой пример**

```python
def log(func):
    def wrapper(*args, **kwargs):
        print(f"Вызов {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log
def hello():
    print("Привет!")

hello()
```

**Когда использовать**   

* Логирование
* Кеширование (`functools.lru_cache`)
* Проверка прав доступа
* Время выполнения функций
* Валидация входных данных

**С сохранением метаданных функции**
Используй `functools.wraps`:

```python
from functools import wraps

def dec(func):
    @wraps(func)
    def wrapper(*a, **kw):
        return func(*a, **kw)
    return wrapper
```

**Декораторы с аргументами**

```python
def repeat(n):
    def decorator(func):
        def wrapper(*a, **kw):
            for _ in range(n):
                func(*a, **kw)
        return wrapper
    return decorator

@repeat(3)
def hi():
    print("hi")
```

**Декорирование методов классов** 
Декораторы работают одинаково, но учитывай `self`: 

```python
def log(func):
    def wrapper(self, *a, **kw):
        print(func.__name__)
        return func(self, *a, **kw)
    return wrapper
```

### Аннотации типов (type hints) <a id="type-hints"></a>

Аннотации типов помогают явно указывать типы переменных, аргументов и возвращаемых значений. Они не влияют на выполнение, но улучшают читаемость и работу линтеров/IDE.

**Базовые примеры**

```python
a: int = 10
b: str = "text"

def add(x: int, y: int) -> int:
    return x + y
```

**Коллекции**

```python
from typing import List, Tuple, Dict, Set

nums: List[int] = [1, 2, 3]
pair: Tuple[str, int] = ("age", 26)
data: Dict[str, float] = {"x": 1.5}
unique: Set[str] = {"a", "b"}
```

**Объединение типов**

```python
from typing import Union

def parse(value: Union[int, str]) -> str:
    return str(value)
```

В новых версиях можно короче:

```python
def parse(value: int | str) -> str:
    return str(value)
```

**Необязательные параметры**

```python
from typing import Optional

def get_name(name: Optional[str]) -> str:
    return name or "Unknown"
```

**Any, None, NoReturn**

```python
from typing import Any, NoReturn

value: Any = "что угодно"

def empty() -> None:
    pass
```

### Классы и объекты <a id="классы-и-объекты"></a>

Класс — это шаблон (чертёж) для создания объектов.
Объект — это экземпляр класса с собственными данными и поведением.

**Создание класса**

```python
class User:
    pass
```

**Создание объекта**

```python
u = User()
```

**Атрибуты объекта**

```python
class User:
    def __init__(self, name, age):
        self.name = name
        self.age = age

u = User("Nikolai", 26)
```

**Методы**
Функции внутри класса, работающие с объектом.

```python
class Car:
    def drive(self):
        print("Едем")
```

**Атрибуты класса**
Общие для всех объектов.

```python
class A:
    version = 1.0
```

**Атрибуты объекта**
Уникальны для экземпляров (создаются в `__init__`):

```python
self.data = ...
```

**Разница между классом и объектом**

* Класс определяет *что такое* объект.
* Объект — конкретный экземпляр с заполненными данными.

**Основная идея**
Класс объединяет *данные* (атрибуты) и *логику* (методы) в одну сущность.

### Инкапсуляция и свойства (property) <a id="инкапсуляция"></a>

Инкапсуляция — это механизм сокрытия внутренних данных объекта и контроля доступа к ним. В Python она реализуется не строго, а *конвенционально*, но достаточно гибко.

**Уровни доступа в Python**

* `_protected` — один нижний прочерк.
  Сигнализирует, что атрибут «внутренний», но доступ не запрещён.
* `__private` — два нижних подчёркивания.
  Запускает *name mangling*: имя превращается в `_ИмяКласса__атрибут`, что защищает от случайного доступа.

```python
class User:
    def __init__(self, name, age):
        self._age = age          # защищённое поле
        self.__token = "abc123"  # приватное поле
```

**Почему нет строгой инкапсуляции**
Python следует принципу *“We are all consenting adults”* — можно получить доступ, но считается плохой практикой.

```python
user = User("N", 25)
print(user._age)                  # можно, но не рекомендуется
print(user._User__token)          # доступ к приватному через name mangling
```

---

**Свойства (`property`) — python-способ безопасной инкапсуляции**    

`property` позволяет управлять доступом к полю: читать, изменять, делать валидацию — при этом синтаксис остаётся как у обычного атрибута.

```python
class Person:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):                # getter
        return self._age

    @age.setter
    def age(self, value):         # setter
        if value < 0:
            raise ValueError("Возраст не может быть отрицательным")
        self._age = value

    @age.deleter
    def age(self):                # deleter (редко нужен)
        del self._age
```

Использование:

```python
p = Person(20)
print(p.age)      # getter
p.age = 30        # setter
```

---

**Когда использовать `property`**

* нужна валидация при изменении данных;
* нужно скрыть внутреннюю реализацию, сохранив удобный API;
* нужно вычислять значение "на лету";
* заменить публичное поле на управляемый доступ, не ломая код.

**Альтернативы**

* `@cached_property` — ленивые вычисления;
* `attrs`, `pydantic` — автоматизация проверки и хранения данных.

### Наследование и полиморфизм <a id="наследование-и-полиморфизм"></a>

Наследование позволяет одному классу брать поведение и свойства другого, а полиморфизм — использовать одинаковый интерфейс для разных реализаций.

**Наследование**

* Дочерний класс получает атрибуты и методы родительского.
* Можно расширять или переопределять методы.
* `super()` позволяет вызвать методы родителя.

```python
class Animal:
    def speak(self):
        return "..."

class Dog(Animal):
    def speak(self):
        return "Гав"

d = Dog()
print(d.speak())  # Гав
```

**super()**

```python
class A:
    def run(self):
        print("A run")

class B(A):
    def run(self):
        super().run()
        print("B run")
```

**Виды наследования**

* Одиночное
* Множественное

```python
class A: ...
class B: ...
class C(A, B): ...
```

* Многоуровневое
* Иерархическое

**Порядок разрешения методов (MRO)**
Python использует алгоритм C3 linearization.
Порядок можно посмотреть:

```python
print(C.mro())
```

---

**Полиморфизм**
Одинаковый вызов — разное поведение в зависимости от класса.

```python
class Cat:
    def speak(self): return "Мяу"

class Dog:
    def speak(self): return "Гав"

def say(animal):
    print(animal.speak())

say(Cat())
say(Dog())
```

**Полиморфизм через переопределение методов**
Дочерний класс меняет реализацию метода родителя.

**Паттерн "утка" (duck typing)**
Python не требует общего базового класса — важны поведение и наличие нужных методов.

```python
class Human:
    def speak(self): return "Hello"

say(Human())  # тоже работает
```

**Абстракция (через ABC — для строгого полиморфизма)**

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

class Square(Shape):
    def area(self):
        return 4 * 4
```

### Абстрактные классы (ABC) <a id="абстрактные-классы"></a>

Абстрактные классы используются для создания общего шаблона поведения, который должен быть реализован в дочерних классах. Они определяют **абстрактные методы** — методы без реализации, которые обязательно нужно переопределить.

**Зачем нужны**

* Задают строгий контракт для наследников
* Помогают проектировать архитектуру
* Предотвращают создание объектов «незавершённых» классов
* Удобны для интерфейсов и общего API

**Реализация через `abc`**  

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass
```

Нельзя создать объект абстрактного класса:

```python
Shape()  # ошибка
```

**Реализация в дочерних**

```python
class Circle(Shape):
    def __init__(self, r):
        self.r = r

    def area(self):
        return 3.14 * self.r ** 2
```

**Абстрактные свойства**

```python
class Base(ABC):
    @property
    @abstractmethod
    def value(self):
        pass
```

**Конкретные методы в ABC**
Можно смешивать абстрактные и обычные методы.

```python
class Example(ABC):
    @abstractmethod
    def run(self):
        pass

    def info(self):
        return "Конкретная логика"
```

**Абстрактные классы vs интерфейсы**
В Python интерфейсы реализуются теми же ABC —
интерфейс = ABC, содержащий только абстрактные методы.

### Магические методы (dunder) <a id="магические-методы"></a>

Магические методы — это специальные методы, имена которых начинаются и заканчиваются двумя подчёркиваниями (`__method__`). Они позволяют объектам вести себя как встроенные типы: участвовать в операторных выражениях, быть итерируемыми, сравниваемыми и т.д.

**Основная идея**
Магические методы перехватывают операции языка и дают возможность определить собственную логику для операторов, конструкторов, протоколов (итерация, контекстный менеджер и пр.).

**Категории магических методов**

**Инициализация и создание**

```python
__new__(cls, ...)   # создание объекта (редко переопределяется)
__init__(self, ...) # инициализация
__del__(self)       # финализатор (использовать осторожно)
```

**Строковое представление**

```python
__str__(self)  # human-readable
__repr__(self) # точное, отладочное представление
```

**Операторы сравнения**

```python
__eq__(self, other)  # ==
__ne__(self, other)  # !=
__lt__(self, other)  # <
__gt__(self, other)  # >
__le__(self, other)  # <=
__ge__(self, other)  # >=
```

**Арифметические операторы**

```python
__add__(self, other)      # +
__sub__(self, other)      # -
__mul__(self, other)      # *
__truediv__(self, other)  # /
__floordiv__(self, other) # //
__mod__(self, other)      # %
__pow__(self, other)      # **
```

**Операторы in-place**  

```python
__iadd__(self, other)  # +=
__imul__(self, other)  # *=
```

**Хеширование**

```python
__hash__(self)
```

**Контейнеры**

```python
__len__(self)
__getitem__(self, key)
__setitem__(self, key, value)
__delitem__(self, key)
__contains__(self, item)
```

**Итерация**

```python
__iter__(self)  # возвращает итератор
__next__(self)  # логика следующего элемента
```

**Контекстный менеджер (`with`)** 

```python
__enter__(self)
__exit__(self, exc_type, exc, tb)
```

**Вызов объекта как функции**

```python
__call__(self, ...)
```

**Простой пример**

```python
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)
print(v1 + v2)  # Vector(4, 6)
```

### Множественное наследование <a id="множественное-наследование"></a>

Множественное наследование позволяет классу наследовать поведение сразу от нескольких родительских классов. Это даёт гибкость, но требует аккуратности из-за возможных конфликтов методов.

**Как работает**

* Класс может указывать несколько базовых классов через запятую.
* Порядок важен: Python использует `MRO` (Method Resolution Order) — порядок поиска атрибутов.

**Пример**

```python
class A:
    def hello(self):
        print("A")

class B:
        def hello(self):
            print("B")

class C(A, B):
    pass

C().hello()  # A (первый в MRO)
```

**MRO (порядок разрешения методов)**
Определяет последовательность поиска атрибутов.
Получить можно через:

```python
print(C.mro())
```

**Конфликты имён**
Если в разных родителях одноимённые методы — будет вызван первый по MRO.

**Использование super()**
При множественном наследовании super вызывает следующий класс по MRO.

```python
class A:
    def run(self):
        print("A")
        super().run()

class B:
    def run(self):
        print("B")
        super().run()

class C(A, B):
    def run(self):
        print("C")
        super().run()

class D:
    def run(self):
        print("D")

class X(C, D):
    pass

X().run()
```
 
**Когда применять**  

* Миксины (добавление функционала без основной логики)
* Разделение поведения на независимые компоненты

**Паттерн mixin**
Класс, который предоставляет дополнительные методы, но не является самостоятельной сущностью.

```python
class LogMixin:
    def log(self, msg):
        print(f"[LOG]: {msg}")

class Service(LogMixin):
    pass

s = Service()
s.log("hello")
```

## ❌ Работа с исключениями <a id="работа-с-исключениями"></a>

### Иерархия исключений <a id="иерархия-исключений"></a>

Иерархия исключений в Python построена вокруг базового класса `BaseException`. Все остальные ошибки — его наследники. Это нужно, чтобы можно было ловить как конкретные ошибки, так и целые группы.

**Ключевые уровни и классы**

* `BaseException`
  Корневой класс всех исключений. Прямые наследники:

  * `SystemExit` — выход из программы
  * `KeyboardInterrupt` — прерывание Ctrl+C
  * `GeneratorExit` — завершение генератора
  * `Exception` — базовый класс для «обычных» ошибок

* `Exception`
  Главная ветка, которую обычно обрабатывают. Популярные подклассы:

  * `ArithmeticError`

    * `ZeroDivisionError`
    * `OverflowError`
    * `FloatingPointError`
  * `LookupError`

    * `IndexError`
    * `KeyError`
  * `TypeError`
  * `ValueError`
  * `NameError`
  * `AttributeError`
  * `FileNotFoundError`
  * `IOError` / `OSError`
  * `RuntimeError`

**Важно знать**

* Ловить `Exception` — можно, это «нормальные ошибки».
* Ловить `BaseException` — нельзя (почти всегда), потому что поймаете `SystemExit` и `KeyboardInterrupt`.
* Иерархия позволяет перехватывать более широкий класс ошибок:

  ```python
  try:
      d = {}
      print(d["x"])
  except LookupError:  # перехватит KeyError и IndexError
      print("Ошибка поиска")
  ```

**Пользовательские исключения**

Создаются наследованием от `Exception`:

```python
class MyError(Exception):
    pass
```

**Рекомендации**

* Наследуйтесь от `Exception`, а не от `BaseException`.
* Делайте свои исключения информативными.
* Используйте «узкие» except, а не `except Exception:` без надобности.

### try / except / else / finally <a id="try-except"></a>

try / except / else / finally — механизм обработки исключений в Python.

try: блок с кодом, который может вызвать исключение.  
except: перехватывает и обрабатывает указанное исключение.  
else: выполняется только если ошибок не возникло.  
finally: выполняется всегда (например, закрытие файла, освобождение ресурсов).

Пример:
try:
    x = int(input())
except ValueError:
    print("Ошибка: введено не число")
else:
    print("Корректный ввод")
finally:
    print("Завершение работы")

Схема работы:
1) выполняется try  
2) если возникла ошибка → переход в except   
3) если ошибок нет → выполняется else  
4) finally запускается в любом случае

Можно указывать разные except для разных типов ошибок:
try:
    f = open("data.txt")
    n = int(input())
except FileNotFoundError:
    print("Файл не найден")
except ValueError:
    print("Некорректный ввод")

### Контекстные менеджеры (with) <a id="контекстные-менеджеры"></a>

```
Контекстные менеджеры управляют ресурсами, гарантируя их корректное открытие и закрытие через with.

with open("file.txt") as f:
    data = f.read()

Эквивалентно:
f = open("file.txt")
try:
    data = f.read()
finally:
    f.close()

Создание собственного менеджера через класс:
class MyManager:
    def __enter__(self):
        print("Вход в контекст")
        return self
    def __exit__(self, exc_type, exc_value, traceback):
        print("Выход из контекста")
        return False  # если True — подавляет исключения

with MyManager() as m:
    print("Внутри блока")

Можно использовать декоратор contextlib.contextmanager для генераторов:
from contextlib import contextmanager

@contextmanager
def my_context():
    print("Начало")
    yield
    print("Конец")

with my_context():
    print("Внутри блока")

Преимущества:

3) компактный и читаемый код
```

## 🗂️ Коллекции и структуры данных <a id="коллекции"></a>

### List, Tuple, Set, Dict <a id="list-tuple-set-dict"></a>

**Коллекции в Python**

| Тип   | Изменяемость | Порядок | Дубли           | Пример создания     | Основные методы                                                                |
| ----- | ------------ | ------- | --------------- | ------------------- | ------------------------------------------------------------------------------ |
| List  | Да           | Да      | Да              | `lst = [1,2,3]`     | `append()`, `extend()`, `pop()`, `insert()`, `remove()`, `sort()`, `reverse()` |
| Tuple | Нет          | Да      | Да              | `tup = (1,2,3)`     | `count()`, `index()`                                                           |
| Set   | Да           | Нет     | Нет             | `s = {1,2,3}`       | `add()`, `remove()`, `discard()`, `union()`, `intersection()`, `difference()`  |
| Dict  | Да           | Да*     | Ключи уникальны | `d = {'a':1,'b':2}` | `keys()`, `values()`, `items()`, `get()`, `pop()`, `update()`                  |

**Особенности**  

* List: хранит элементы любого типа, допускает вложенные списки.
* Tuple: неизменяемый, используется для констант или как ключи в dict.
* Set: уникальные элементы, неупорядоченный, эффективен для проверки вхождения.
* Dict: хранит пары ключ-значение, ключи уникальны, начиная с Python 3.7 порядок вставки сохраняется.

**Примеры**

```python
# List
lst = [1, 2, 3]
lst.append(4)        # [1, 2, 3, 4]
lst.remove(2)        # [1, 3, 4]

# Tuple
tup = (1, 2, 3)
print(tup[0])        # 1

# Set
s = {1, 2, 3}
s.add(4)             # {1, 2, 3, 4}
s.remove(2)          # {1, 3, 4}

# Dict
d = {'a': 1, 'b': 2}
d['c'] = 3           # {'a':1, 'b':2, 'c':3}
d.get('b')           # 2
```

### collections: Counter, deque, defaultdict, namedtuple <a id="collections-модуль"></a>

**Модуль collections** — расширяет стандартные коллекции Python.

| Класс       | Описание                                                      | Пример создания                                                                                      | Основные методы / особенности                                 |
| ----------- | ------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| Counter     | Счётчик объектов, словарь для подсчёта элементов              | `from collections import Counter`<br>`c = Counter([1,2,2,3])`                                        | `most_common(n)`, `elements()`, поддержка +, -                |
| deque       | Двусторонняя очередь, быстрая вставка/удаление с обоих концов | `from collections import deque`<br>`d = deque([1,2,3])`                                              | `append()`, `appendleft()`, `pop()`, `popleft()`, `rotate(n)` |
| defaultdict | Словарь с значением по умолчанию                              | `from collections import defaultdict`<br>`dd = defaultdict(int)`                                     | `dd[key]` вернёт default если ключ отсутствует                |
| namedtuple  | Кортеж с именованными полями                                  | `from collections import namedtuple`<br>`Point = namedtuple('Point', ['x','y'])`<br>`p = Point(1,2)` | Доступ через `p.x`, `p.y`, неизменяемый                       |

**Примеры использования**  

```python
from collections import Counter, deque, defaultdict, namedtuple

# Counter
c = Counter('abracadabra')
print(c.most_common(2))  # [('a', 5), ('b', 2)]

# deque
d = deque([1,2,3])
d.appendleft(0)           # deque([0,1,2,3])
d.pop()                   # deque([0,1,2])

# defaultdict
dd = defaultdict(int)
dd['apples'] += 1         # defaultdict(int, {'apples': 1})

# namedtuple
Point = namedtuple('Point', ['x','y'])
p = Point(10, 20)
print(p.x, p.y)           # 10 20
```

### Хэш-таблицы и уникальные ключи <a id="хэш-таблицы"></a>

**Хэш-таблицы** — структуры данных, которые используют **ключи** для быстрого доступа к значениям.

**Основные свойства**

| Свойство                      | Описание                                                |
| ----------------------------- | ------------------------------------------------------- |
| Ключи уникальны               | Каждый ключ встречается только один раз                 |
| Быстрый доступ                | Получение, вставка и удаление — O(1) в среднем          |
| Использует хэш-функцию        | Хэш ключа определяет место хранения значения            |
| Поддержка неизменяемых ключей | Ключи должны быть неизменяемыми: числа, строки, кортежи |

**Примеры хэш-таблиц в Python**  

| Коллекция | Описание                                          | Пример                    |
| --------- | ------------------------------------------------- | ------------------------- |
| dict      | Стандартный словарь, ключи уникальны              | `d = {'a':1, 'b':2}`      |
| set       | Множество уникальных элементов                    | `s = {1,2,3}`             |
| frozenset | Неизменяемое множество для использования как ключ | `fs = frozenset([1,2,3])` |

**Пример работы с dict и set**

```python
# dict
d = {'apple': 5, 'banana': 3}
d['orange'] = 7          # добавление

```

**Особенности**

* Ключи dict и элементы set должны быть **хэшируемыми** (immutable: int, str, tuple с immutable элементами).
* Повторные ключи перезаписывают значение.
* dict сохраняет порядок вставки начиная с Python 3.7.

### Очереди и стеки <a id="очереди-и-стеки"></a> 

| Структура       | Принцип работы             | Основные операции                                      | Python реализация                                  |
| --------------- | -------------------------- | ------------------------------------------------------ | -------------------------------------------------- |
| Стек (Stack)    | LIFO — Last In, First Out  | push (добавить), pop (удалить), peek (посмотреть верх) | list (`append()`, `pop()`) или `collections.deque` |
| Очередь (Queue) | FIFO — First In, First Out | enqueue (добавить), dequeue (удалить), peek            | `collections.deque` (`append()`, `popleft()`)      |

**Примеры**

```python
from collections import deque

# Стек
stack = []
stack.append(1)      # push
stack.append(2)
top = stack.pop()    # pop -> 2

# Очередь
queue = deque([1,2,3])
queue.append(4)      # enqueue
first = queue.popleft()  # dequeue -> 1
```

**Особенности**  

* Стек удобно использовать для **обратного прохода**, рекурсивных алгоритмов, undo/redo.
* Очередь подходит для **обработки задач по порядку**, BFS, симуляции процессов.
* `deque` эффективнее `list` для операций с обоих концов: O(1) vs O(n).

## 🟤 Современные возможности Python <a id="современные-возможности"></a>

### Генераторы и yield <a id="генераторы-и-yield"></a>

**Генераторы** — специальные итераторы, которые создают значения «на лету», экономя память.

| Особенность        | Описание                                     |
| ------------------ | -------------------------------------------- |
| Ленивые вычисления | Значения создаются по мере обращения         |
| Итератор           | Поддерживают `next()` и циклы `for`          |
| Экономия памяти    | Не хранят весь результат в памяти, как list  |
| Синтаксис          | Функции с `yield` или генераторные выражения |

**Создание генератора с yield**

```python
def gen_numbers(n):
    for i in range(n):
        yield i  # возвращает значение, но сохраняет состояние функции

g = gen_numbers(5)
print(next(g))  # 0
print(list(g))  # [1,2,3,4]
```

**Генераторное выражение (аналог спискового, но лениво)**

```python
squares = (x**2 for x in range(5))
print(next(squares))  # 0
print(list(squares))  # [1,4,9,16]
```

**Преимущества генераторов**

* Позволяют работать с **большими потоками данных** без перегрузки памяти.
* Упрощают код, заменяя сложные классы-итераторы.
* Можно строить **цепочки генераторов** для фильтрации и преобразования данных.

### Декораторы функций и классов <a id="декораторы-функций-и-классов"></a>

**Декоратор** — это функция или класс, который **оборачивает другую функцию или класс**, добавляя функциональность без изменения исходного кода.

| Тип декоратора    | Описание                                                                        | Теоретическая суть                                                                                                                     |
| ----------------- | ------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Функция-декоратор | Принимает функцию, возвращает новую функцию                                     | Оборачивает функцию, создавая промежуточную функцию `wrapper`, которая выполняет дополнительный код до и после вызова исходной функции |
| Класс-декоратор   | Класс с `__call__`, который оборачивает функцию                                 | Создает объект-декоратор, который при вызове имитирует функцию и может модифицировать поведение вызова                                 |
| Декоратор метода  | Используется для методов классов (`@staticmethod`, `@classmethod`, `@property`) | Изменяет поведение метода, например делает его статическим, класс-методом или свойством класса                                         |

**Примеры использования** 

```python
# Декоратор функции
def debug(func):
    def wrapper(*args, **kwargs):
        print(f"Вызов {func.__name__} с {args}, {kwargs}")
        return func(*args, **kwargs)
    return wrapper

@debug
def add(a, b):
    return a + b

print(add(2,3))
```

**Преимущества декораторов**

* Разделение функциональности (логирование, проверка прав, кэширование).
* Повторное использование обёрток для разных функций или классов.
* Поддержка цепочек декораторов: `@dec1 @dec2 def func(): ...`

### Контекстные менеджеры <a id="контекстные-менеджеры-с"></a>

**Контекстный менеджер** — объект, который управляет ресурсами с помощью конструкции `with`, гарантируя корректное освобождение ресурсов.

| Особенность                    | Описание                                                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| Управление ресурсами           | Автоматически открывает и закрывает файлы, соединения, блокировки и т.д.                                                  |
| Методы                         | `__enter__()` — подготовка ресурса<br>`__exit__(exc_type, exc_val, exc_tb)` — освобождение ресурса и обработка исключений |
| Применение                     | Используется через `with`, уменьшает количество ошибок при работе с ресурсами                                             |
| Поддержка вложенных менеджеров | Можно использовать несколько `with` в одной конструкции                                                                   |

**Примеры**  

```python
# Контекстный менеджер для файла
with open("file.txt", "w") as f:
    f.write("Hello, world!")  # файл автоматически закроется после блока

# Создание собственного контекстного менеджера через класс
class ManagedResource:
    def __enter__(self):
        print("Ресурс открыт")
        return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        print("Ресурс закрыт")
        return False  # ошибки не подавляются

with ManagedResource() as r:
    print("Используем ресурс")
```

**Создание через декоратор `contextlib.contextmanager`**

```python
from contextlib import contextmanager

@contextmanager
def managed_resource():
    print("Ресурс открыт")
    yield
    print("Ресурс закрыт")

with managed_resource():
    print("Используем ресурс")
```

**Преимущества**

* Гарантированное освобождение ресурсов.
* Чистый и читаемый код.
* Упрощает обработку исключений и ошибок при работе с ресурсами.

### Аннотации типов (typing) <a id="typing"></a>

**Аннотации типов** — механизм для указания ожидаемого типа переменных, аргументов и возвращаемого значения функций.

| Особенность             | Описание                                                            |
| ----------------------- | ------------------------------------------------------------------- |
| Статическая проверка    | Позволяет инструментам (mypy, PyCharm) проверять корректность типов |
| Не влияет на выполнение | Python игнорирует типы во время выполнения                          |
| Читаемость кода         | Улучшает документацию и понимание кода                              |
| Поддержка сложных типов | `List`, `Dict`, `Tuple`, `Optional`, `Union`, `Callable`            |

**Примеры**

```python
from typing import List, Dict, Optional, Union, Tuple

# Простая аннотация функции
def add(a: int, b: int) -> int:
    return a + b

# Списки и словари
def process_items(items: List[str]) -> Dict[str, int]:
    return {item: len(item) for item in items}

# Опциональный аргумент
def greet(name: Optional[str] = None) -> str:
    return f"Hello, {name or 'Guest'}"

# Union и Tuple
def handle_data(data: Union[int, str]) -> Tuple[str, int]:
    if isinstance(data, int):
        return ("int", data)
    return ("str", len(data))
```

**Преимущества**

* Повышает **надёжность кода** и облегчает поддержку.
* Удобно для **командной разработки** и автодополнения в IDE.
* Совместимо с **статическим анализом и генерацией документации**.

*Основные типы из модуля `typing`*


| Тип                     | Описание                                                      | Пример использования                   |
| ----------------------- | ------------------------------------------------------------- | -------------------------------------- |
| `List[T]`               | Список элементов типа `T`                                     | `names: List[str] = ["Alice", "Bob"]`  |
| `Dict[K, V]`            | Словарь с ключами типа `K` и значениями типа `V`              | `ages: Dict[str, int] = {"Alice": 25}` |
| `Tuple[T1, T2, ...]`    | Кортеж фиксированной длины с типами элементов                 | `point: Tuple[int, int] = (10, 20)`    |
| `Set[T]`                | Множество элементов типа `T`                                  | `unique: Set[int] = {1, 2, 3}`         |
| `Optional[T]`           | Значение типа `T` или `None`                                  | `name: Optional[str] = None`           |
| `Union[T1, T2]`         | Значение может быть одного из нескольких типов                | `data: Union[int, str] = 5`            |
| `Any`                   | Любой тип                                                     | `value: Any = "text"`                  |
| `Callable[[T1, T2], R]` | Функция с аргументами типов `T1, T2` и возвращаемым типом `R` | `func: Callable[[int, int], int]`      |
| `Iterable[T]`           | Объект, по которому можно итерироваться                       | `items: Iterable[str]`                 |
| `Iterator[T]`           | Итератор, который возвращает элементы типа `T`                | `it: Iterator[int]`                    |
| `Type[T]`               | Тип объекта `T`                                               | `cls: Type[MyClass]`                   |
| `Literal`               | Конкретное фиксированное значение                             | `mode: Literal["r","w"]`               |

**Примечания**

* `T`, `K`, `V`, `R` — это **плейсхолдеры для типов** (generics).
* Можно комбинировать: `Optional[Union[int, str]]` — число, строка или `None`.
* Используется для **проверки типов, автодополнения и документации**.

### Pattern Matching (match-case) <a id="pattern-matching"></a>

**Pattern Matching** — новый синтаксис в Python 3.10+, аналог switch/case в других языках, позволяющий сопоставлять структуры данных с шаблонами.

| Особенность                | Описание                                               |
| -------------------------- | ------------------------------------------------------ |
| Сопоставление с образцом   | Проверяет значение и структуру объекта                 |
| Поддержка сложных структур | Можно сопоставлять списки, кортежи, словари, классы    |
| Присвоение переменной      | `as` позволяет сохранять значение в переменную         |
| Поддержка “_”              | Используется как “wildcard” для игнорирования значения |

**Синтаксис и примеры**

```python
# Простейшее сопоставление
value = 2
match value:
    case 1:
        print("Один")
    case 2:
        print("Два")
    case _:
        print("Другое")

*Сопоставление с кортежем*

point = (0, 0)
match point:
    case (0, 0):
        print("Начало координат")
    case (x, 0):
        print(f"На оси X: {x}")
    case (0, y):
        print(f"На оси Y: {y}")
    case (x, y):
        print(f"Точка ({x},{y})")

*Сопоставление с классом*

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(1, 0)
match p:
    case Point(0, 0):
        print("Начало координат")
    case Point(x, y):
        print(f"Точка ({x},{y})")
```

**Преимущества**

* Более **читабельный и декларативный код** для ветвления.
* Позволяет обрабатывать **сложные структуры данных** без вложенных `if-else`.
* Поддерживает **деструктуризацию**, присвоение переменным и игнорирование значений.

## 🔵 Модули и управление пространством имён <a id="модули-и-пространство-имён"></a>

### Импорт и создание модулей <a id="импорт-и-модули"></a>

**Модули** — это файлы `.py`, содержащие функции, классы и переменные, которые можно импортировать и использовать в других программах.

| Операция                   | Описание                                     | Пример                                                                                        |
| -------------------------- | -------------------------------------------- | --------------------------------------------------------------------------------------------- |
| Импорт всего модуля        | Подключает модуль целиком                    | `import math`<br>`math.sqrt(16)`                                                              |
| Импорт конкретных объектов | Подключает только нужные функции или классы  | `from math import sqrt, pi`<br>`sqrt(16)`                                                     |
| Псевдоним (alias)          | Дает модулю или объекту удобное имя          | `import numpy as np`<br>`np.array([1,2,3])`                                                   |
| Импорт всех объектов       | Импортирует всё из модуля (не рекомендуется) | `from math import *`                                                                          |
| Создание своего модуля     | Любой `.py` файл с функциями и классами      | `# файл mymodule.py`<br>`def greet(): print("Hi")`<br>`import mymodule`<br>`mymodule.greet()` |

**Особенности**

* Модули создают **пространство имён**, чтобы избежать конфликтов имен.
* Можно использовать вложенные модули и пакеты (папки с `__init__.py`).
* Стандартная библиотека Python содержит сотни готовых модулей (`os`, `sys`, `math`, `random`, `datetime`).

**Пример использования пользовательского модуля**

```python
# mymodule.py
def hello(name):
    print(f"Hello, {name}!")

# main.py
import mymodule
mymodule.hello("Alice")  # Hello, Alice!
```

### Модуль `__main__` и точка входа <a id="main"></a>

Модуль `__main__` — это специальный модуль, который становится **точкой входа**, когда Python запускает файл напрямую. Интерпретатор присваивает переменной `__name__` значение `__main__`, и по этому признаку можно определить, был ли модуль запущен как программа или импортирован.

| Термин                       | Суть                                                     |
| :--------------------------- | :------------------------------------------------------- |
| `__main__`                   | Имя текущего запускаемого модуля                         |
| Точка входа                  | Код, выполняющийся при запуске приложения                |
| `if __name__ == "__main__":` | Условие для запуска кода только при прямом запуске файла |

Основные идеи:

* При запуске файла Python выставляет `__name__ = "__main__"`.     
* При импорте — `__name__ = "<имя_модуля>"`, поэтому код внутри условия не выполнится.
* Используется для разделения библиотечного кода и логики запуска.

Пример использования:

```
def run():
    print("Старт")

if __name__ == "__main__":
    run()
```

Если запустить `python app.py`, выполняется `run()`.
Если импортировать `import app`, блок не запускается.

Файл `__main__.py` внутри пакета позволяет запускать пакет командой:

```
python -m package_name  
```

Пример `package_name/__main__.py`:

```
print("Пакет запущен")
```

Зачем использовать:

* определение входной точки программы
* предотвращение лишнего кода при импорте
* организация структуры проектов и CLI-инструментов

## 👑 Управление памятью и объектами <a id="память"></a>

### Модель памяти Python <a id="модель-памяти"></a>

Модель памяти Python описывает, как интерпретатор хранит объекты, управляет их временем жизни и работает со ссылками. Всё в Python является объектом, и каждый объект имеет адрес в памяти, тип и счётчик ссылок.

| Термин                | Значение                                                  |
| :-------------------- | :-------------------------------------------------------- |
| Объект                | Любая сущность в памяти: число, строка, список, функция   |
| Ссылка                | Указатель на объект                                       |
| Идентификатор объекта | Адрес объекта в памяти (`id()`)                           |
| Счётчик ссылок        | Количество переменных, указывающих на объект              |
| Менеджер памяти       | Подсистема, управляющая выделением и освобождением памяти |

Python использует **двухуровневую систему управления памятью**:

1. подсчёт ссылок
2. сборщик мусора (GC) для циклических ссылок

Основные принципы модели:

* переменная — это не контейнер, а *ссылка* на объект
* объект существует, пока на него есть хотя бы одна ссылка
* когда количество ссылок становится 0, объект автоматически удаляется
* все сложные объекты хранятся в куче (heap)
* маленькие числа и короткие строки кэшируются для оптимизации

Пример поведения ссылок:

```
a = [1, 2]
b = a     # теперь две ссылки на один объект
a.append(3)
print(b)  # [1, 2, 3]
```

Обе переменные указывают на один и тот же список.

Счётчик ссылок можно проверить через модуль `sys`:

```
import sys
x = []
sys.getrefcount(x)
```

Сборщик мусора (GC) работает на основе поколений:

| Поколение | Суть                                     |
| :-------- | :--------------------------------------- |
| 0         | новые объекты, часто проверяются         |
| 1         | пережившие некоторое количество проверок |
| 2         | долго живущие объекты, проверяются редко |

Python периодически очищает циклические ссылки, которые не могут быть удалены подсчётом ссылок напрямую.

Типы памяти внутри Python:

| Область       | Что хранится                               |
| :------------ | :----------------------------------------- |
| Stack         | локальные переменные функций (ссылки)      |
| Heap          | объекты (списки, словари, строки, функции) |
| Static memory | встроенные объекты, кэш чисел и строк      |

Ключевые особенности модели памяти Python:

* автоматическое управление памятью
* подсчёт ссылок как основной механизм
* GC как дополнительный механизм
* неизменяемые объекты (int, str, tuple) создают новые экземпляры при изменениях
* изменяемые объекты (list, dict, set) меняют состояние по ссылке

Модель памяти помогает понимать такие явления, как:

* мутабельность и иммутабельность
* копирование объектов
* поведение аргументов функций
* утечки памяти из-за циклических ссылок

### Сборщик мусора (Garbage Collector) <a id="gc"></a>

Сборщик мусора (Garbage Collector, GC) — это механизм Python, который автоматически освобождает память, удаляя объекты, которые больше не используются. Он работает вместе с подсчётом ссылок и предназначен для очистки циклических ссылок.

| Термин                      | Суть                                                                   |
| :-------------------------- | :--------------------------------------------------------------------- |
| Подсчёт ссылок              | Основной механизм: объект удаляется, когда ссылок на него становится 0 |
| Циклическая ссылка          | Ситуация, когда объекты ссылаются друг на друга, мешая удалению        |
| GC                          | Дополнительный механизм, который находит и удаляет циклы               |
| Поколения (generations)     | Группы объектов по "возрасту"                                          |
| Корневая причина наличия GC | Подсчёт ссылок не способен удалить циклы                               |

Как Python управляет памятью:

| Этап | Описание |
| :--- | :------- |

1. Подсчёт ссылок | При каждом создании/удалении ссылки счётчик обновляется
2. Удаление объекта | Когда счётчик = 0, объект уничтожается
3. Запуск GC | Периодически ищет циклические ссылки и удаляет их

GC использует модель поколений:

| Поколение | Особенности                              |
| :-------- | :--------------------------------------- |
| 0         | Новые объекты, проверяются чаще всего    |
| 1         | Средний “возраст”, реже проверяются      |
| 2         | Долго живущие объекты, проверяются редко |

Схема работы поколений:

* при создании объект находится в поколении 0
* если пережил несколько сборок, перемещается в следующее поколение
* считается, что долгоживущие объекты не образуют циклы

Когда GC нужен:

* работа со сложными структурами данных (списки списков, графы, деревья)
* использование замыканий и объектов, содержащих ссылки друг на друга

Пример реальной циклической ссылки:

```
a = []
b = []
a.append(b)
b.append(a)
```

Счётчики ссылок не дадут этим объектам удалиться, но GC их обнаружит.

Работа со сборщиком мусора:

| Команда              | Назначение                            |
| :------------------- | :------------------------------------ |
| `import gc`          | Импорт модуля                         |
| `gc.collect()`       | Принудительно запустить сборку мусора |
| `gc.get_stats()`     | Статистика по поколениям              |
| `gc.get_threshold()` | Порог запуска GC                      |

Почему важно понимать GC:

* помогает избегать утечек памяти
* важно при работе с большими структурами данных
* даёт понимание, почему объект не удаляется сразу
* помогает оптимизировать производительность

### Подсчёт ссылок и циклические ссылки <a id="refcount"></a>

Подсчёт ссылок — основной механизм управления памятью в Python. Каждый объект хранит число ссылок на себя. Когда это число становится 0, объект удаляется сразу. Проблема возникает при циклических ссылках — когда объекты ссылаются друг на друга, и их нельзя удалить простым подсчётом ссылок.

| Термин             | Суть                                      |
| :----------------- | :---------------------------------------- |
| Ссылка             | Указатель переменной на объект            |
| Счётчик ссылок     | Количество ссылок на объект               |
| Циклическая ссылка | Объекты ссылаются друг на друга           |
| Утечка из-за цикла | Счётчик ссылок никогда не становится 0    |
| GC                 | Механизм, который находит и удаляет циклы |

Как работает подсчёт ссылок:

| Ситуация                     | Что происходит              |
| :--------------------------- | :-------------------------- |
| Создание переменной          | Счётчик увеличивается       |
| Присвоение другой переменной | Счётчик снова увеличивается |
| Удаление переменной (`del`)  | Счётчик уменьшается         |
| Выход из области видимости   | Счётчик уменьшается         |
| Счётчик стал 0               | Объект удаляется мгновенно  |

Пример:

```
x = [1, 2]   # 1 ссылка
y = x        # 2 ссылки
del x        # 1 ссылка
del y        # 0 ссылок → объект уничтожен
```

Циклические ссылки:

```
a = []
b = []
a.append(b)
b.append(a)
```

| Ситуация                         | Проблема                            |
| :------------------------------- | :---------------------------------- |
| `a` → `b`, `b` → `a`             | У каждого объекта есть ссылка       |
| Счётчик ссылок не снижается до 0 | Обычный механизм не удаляет объекты |
| GC ищет циклы                    | И удаляет их                        |

Почему циклы возникают:

| Причина             | Пример                                |
| :------------------ | :------------------------------------ |
| Структуры данных    | Списки внутри списков                 |
| Графы и деревья     | Вершины ссылаются друг на друга       |
| Замыкания           | Функции хранят внешние переменные     |
| Классы с атрибутами | Объекты содержат ссылки друг на друга |

Как увидеть количество ссылок:

```
import sys
x = []
sys.getrefcount(x)
```

Как избежать циклов:

| Способ                                 | Идея                          |
| :------------------------------------- | :---------------------------- |
| Использовать слабые ссылки (`weakref`) | Ссылка не увеличивает счётчик |
| Освобождать ссылки вручную             | `a.clear()`, `del x`          |
| Не создавать взаимных ссылок           | Избегать круговой структуры   |

Роль GC:

* находит циклы
* удаляет объекты, недоступные через корневые ссылки
* работает по поколениям

### copy() и deepcopy() <a id="deepcopy"></a>

Методы `copy()` и `deepcopy()` используются для копирования объектов. Разница между ними в глубине копирования: поверхностная копия копирует только верхний уровень объекта, а глубокая — всю структуру целиком.

| Термин                              | Суть                                                             |
| :---------------------------------- | :--------------------------------------------------------------- |
| Поверхностная копия (`copy.copy()`) | Создаёт новый объект, но вложенные объекты копируются по ссылкам |
| Глубокая копия (`copy.deepcopy()`)  | Полностью копирует всю структуру, включая вложенные объекты      |
| Ссылка                              | Указатель на объект в памяти                                     |
| Вложенный объект                    | Объект внутри другого объекта                                    |

Поверхностное копирование:

```
import copy

a = [1, [2, 3]]
b = copy.copy(a)
```

| Свойство                     | Поведение            |
| :--------------------------- | :------------------- |
| Верхний уровень              | Копируется           |
| Вложенные списки             | Ссылки сохраняются   |
| Изменение вложенного объекта | Видно в обеих копиях |

Пример:

```
b[1].append(4)
print(a)  # [1, [2, 3, 4]]
```

Глубокое копирование: 

```
import copy

a = [1, [2, 3]]
b = copy.deepcopy(a)
```

| Свойство            | Поведение   |
| :------------------ | :---------- |
| Верхний уровень     | Копируется  |
| Вложенные объекты   | Копируются  |
| Связи между копиями | Отсутствуют |

Пример:

```
b[1].append(4)
print(a)  # [1, [2, 3]]
```

Когда использовать:

| Ситуация                           | Лучший вариант                         |
| :--------------------------------- | :------------------------------------- |
| Объекты простые, без вложенности   | `copy()`                               |
| Списки списков, словари в словарях | `deepcopy()`                           |
| Нужно полностью изолировать копию  | `deepcopy()`                           |
| Требуется производительность       | `copy()` быстрее                       |
| Есть циклические ссылки            | `deepcopy()` справляется автоматически |

Особенности:

| Особенность              | Описание                                           |
| :----------------------- | :------------------------------------------------- |
| Не все объекты копируемы | Некоторые типы запрещают копирование               |
| `deepcopy()` медленнее   | Из-за рекурсивного обхода                          |
| Учитываются циклы        | `deepcopy()` ведёт учёт уже скопированных объектов |
| Мутабельные объекты      | Ведут себя по-разному при изменениях               |

### Изменяемость объектов и ссылки <a id="изменяемость-и-ссылки"></a>

**Изменяемость** — важная особенность, определяющая, можно ли менять объект после его создания.

| Тип объекта      | Изменяемый | Примеры                                        |
| ---------------- | ---------- | ---------------------------------------------- |
| **Изменяемые**   | Да         | `list`, `dict`, `set`, пользовательские классы |
| **Неизменяемые** | Нет        | `int`, `float`, `str`, `tuple`, `frozenset`    |

#### Ссылки на объекты

Все переменные в Python — **ссылки на объекты в памяти**, а не сами данные.

```python
a = [1, 2, 3]
b = a        # b указывает на тот же объект
b.append(4)
print(a)     # [1, 2, 3, 4]
```

| Операция             | Что происходит                                               |
| -------------------- | ------------------------------------------------------------ |
| Присваивание `b = a` | Создается вторая ссылка на тот же объект                     |
| Изменение объекта    | Меняет его для всех ссылок                                   |
| Перезапись `a = ...` | Меняется только ссылка `a`, сам объект остаётся неизменённым |

#### Почему важна изменяемость

| Ситуация                                     | Что важно понимать                                             |
| -------------------------------------------- | -------------------------------------------------------------- |
| Передача в функции                           | Передаётся *ссылка*, а не копия                                |
| Ошибки с изменяемыми значениями по умолчанию | `def f(x=[]): ...` сохраняет состояние между вызовами          |
| Копирование коллекций                        | Нужно использовать `copy()` или `deepcopy()` для независимости |

#### Типовые примеры

```python
# Неизменяемые объекты
x = 10
y = x
y += 1
print(x)  # 10  (создан новый объект для y)

*Изменяемые объекты*
lst1 = [1, 2]
lst2 = lst1
lst2.append(3)
print(lst1)  # [1, 2, 3]

*Избежать совместного изменения*
import copy
lst3 = [1, 2, [3]]
lst4 = copy.deepcopy(lst3)
lst4[2].append(4)
print(lst3)  # [1, 2, [3]]
```

## 🏆 Алгоритмы и вычисления <a id="алгоритмы"></a>

### Рекурсия и итерация <a id="рекурсия"></a>

**Рекурсия и итерация** — два способа многократного выполнения действий.

Подходы

| Подход   | Описание                                                 |
| -------- | -------------------------------------------------------- |
| Рекурсия | Функция вызывает саму себя до достижения базового случая |
| Итерация | Повторение действий с помощью циклов for или while       |

Основные элементы рекурсии

| Элемент         | Назначение                              |
| --------------- | --------------------------------------- |
| Базовый случай  | Условие остановки рекурсии              |
| Рекурсивный шаг | Вызов функции с изменёнными аргументами |
| Стек вызовов    | Хранит состояние каждого вызова функции |

Сравнение рекурсии и итерации

| Критерий             | Рекурсия                          | Итерация                          |
| -------------------- | --------------------------------- | --------------------------------- |
| Читаемость           | Проще для деревьев и графов       | Понятнее для простых задач        |
| Использование памяти | Больше из-за стека вызовов        | Меньше                            |
| Ограничения          | Есть лимит глубины рекурсии       | Ограничений глубины нет           |
| Скорость             | Обычно медленнее                  | Обычно быстрее                    |

**Примеры**

```python
# Итерация
def factorial_iter(n):
    result = 1
    for i in range(1, n + 1):
        result *= i
    return result

# Рекурсия
def factorial_rec(n):
    if n <= 1:          # базовый случай
        return 1
    return n * factorial_rec(n - 1)
````

**Особенности в Python**

| Особенность        | Описание                                      |
| ------------------ | --------------------------------------------- |
| Лимит рекурсии     | Около 1000 вызовов по умолчанию               |
| Хвостовая рекурсия | Не оптимизируется                             |
| Риск ошибок        | Возможен RecursionError при глубокой рекурсии |

Когда что использовать

| Ситуация                     | Подход   |
| ---------------------------- | -------- |
| Работа с деревьями и графами | Рекурсия |
| Простые циклы и подсчёты     | Итерация |
| Ограниченная память          | Итерация |

### Сортировки и поиск <a id="сортировки-и-поиск"></a>

**Сортировки и поиск** — базовые алгоритмы для упорядочивания данных и нахождения элементов.

Сортировки

| Алгоритм        | Идея работы                                   | Сложность (ср.) | Стабильность |
| --------------- | --------------------------------------------- | --------------- | ------------ |
| Bubble Sort     | Обмен соседних элементов                      | O(n²)           | Да           |
| Selection Sort  | Поиск минимума и обмен                        | O(n²)           | Нет          |
| Insertion Sort  | Вставка элемента в отсортированную часть      | O(n²)           | Да           |
| Merge Sort      | Разделяй и властвуй, слияние                  | O(n log n)      | Да           |
| Quick Sort      | Опорный элемент и разбиение                   | O(n log n)      | Нет          |
| Timsort         | Гибрид Merge + Insertion (используется в Python) | O(n log n)  | Да           |

Пример сортировки в Python

```python
data = [5, 2, 9, 1]
data.sort()              # сортировка на месте
sorted_data = sorted(data, reverse=True)
````

**Особенности сортировки в Python**

| Особенность  | Описание                           |
| ------------ | ---------------------------------- |
| Алгоритм     | Timsort                            |
| Стабильность | Да                                 |
| key          | Функция для сортировки по критерию |
| reverse      | Обратный порядок                   |

```python
words = ["apple", "kiwi", "banana"]
words.sort(key=len)      # сортировка по длине строки
```

**Поиск**

| Алгоритм       | Принцип работы            | Условие              | Сложность |
| -------------- | ------------------------- | -------------------- | --------- |
| Линейный поиск | Последовательный перебор  | Любые данные         | O(n)      |
| Бинарный поиск | Деление диапазона пополам | Данные отсортированы | O(log n)  |
| Поиск в hash   | По ключу                  | Уникальные ключи     | O(1)      |

Примеры поиска

```python
# Линейный поиск
def linear_search(arr, x):
    for i, v in enumerate(arr):
        if v == x:
            return i
    return -1

# Бинарный поиск
def binary_search(arr, x):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == x:
            return mid
        elif arr[mid] < x:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

Когда что использовать

| Задача                         | Рекомендация       |
| ------------------------------ | ------------------ |
| Малые наборы данных            | Простые сортировки |
| Большие массивы                | sort / sorted      |
| Частый поиск по ключу          | dict / set         |
| Поиск в отсортированных данных | Бинарный поиск     |

### Сложность алгоритмов (Big O) <a id="big-o"></a>

**Сложность алгоритмов (Big O)** — способ оценки того, как время выполнения или потребление памяти алгоритма растёт при увеличении размера входных данных.

Big O описывает поведение алгоритма в худшем случае и не зависит от конкретного железа или реализации.

Основные обозначения Big O

| Обозначение | Название           | Описание роста                        | Пример |
| ----------- | ------------------ | ------------------------------------- | ------ |
| O(1)        | Константная        | Не зависит от размера входных данных  | Доступ по индексу |
| O(log n)    | Логарифмическая    | Рост замедляется при увеличении n     | Бинарный поиск |
| O(n)        | Линейная           | Пропорциональна размеру данных        | Линейный поиск |
| O(n log n)  | Линейно-логарифм.  | Эффективная сортировка                | Merge Sort |
| O(n²)       | Квадратичная       | Вложенные циклы                       | Bubble Sort |
| O(2ⁿ)       | Экспоненциальная   | Очень быстрый рост                    | Наивная рекурсия |
| O(n!)       | Факториальная      | Крайне неэффективная                  | Перебор всех перестановок | 

Примеры сложности в коде

```python
# O(1)
x = arr[0]

# O(n)
for x in arr:
    print(x)

# O(n²)
for i in arr:
    for j in arr:
        print(i, j)

# O(log n)
def binary_search(arr, x):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == x:
            return mid
        elif arr[mid] < x:
            left = mid + 1
        else:
            right = mid - 1
````

Временная и пространственная сложность

| Тип сложности    | Что измеряет        |
| ---------------- | ------------------- |
| Временная        | Количество операций |
| Пространственная | Используемая память |

Примеры структур данных и сложностей

| Операция / Структура | list  | dict | set  |
| -------------------- | ----- | ---- | ---- |
| Доступ               | O(1)  | O(1) | —    |
| Поиск                | O(n)  | O(1) | O(1) |
| Вставка              | O(1)* | O(1) | O(1) |
| Удаление             | O(n)  | O(1) | O(1) |

*O(1) в среднем, возможны редкие перераспределения памяти*

Как читать Big O

| Что важно помнить                            |
| -------------------------------------------- |
| Константы и младшие члены игнорируются       |
| Оценивается рост, а не точное время          |
| Худший случай важнее среднего                |
| Простота алгоритма не всегда = эффективность |

Когда Big O особенно важно

| Ситуация                        |
| ------------------------------- |
| Большие объёмы данных           |
| Алгоритмы в циклах              |
| Работа с сетями и базами данных |
| Подготовка к собеседованиям     |

###  Основные структуры данных (стек, очередь, дерево) <a id="структуры-данных"></a>

**Основные структуры данных** — это базовые модели организации информации, которые используются для эффективного хранения, поиска и обработки данных.

Стек (Stack)

| Характеристика | Описание |
|----------------|----------|
| Принцип работы | LIFO — последний вошёл, первый вышел |
| Основные операции | push (добавить), pop (удалить), peek (посмотреть верхний элемент) |
| Особенности | Хорош для вложенных вызовов, отмены действий, обработки выражений |
| Использование | Стек вызовов, парсеры, алгоритмы обхода деревьев |

Очередь (Queue)

| Характеристика | Описание |
|----------------|----------|
| Принцип работы | FIFO — первый вошёл, первый вышел |
| Основные операции | enqueue (добавить), dequeue (удалить), front/back |
| Особенности | Обеспечивает порядок обработки элементов |
| Использование | Планировщики задач, буферизация данных, системы обработки событий | 

Дерево (Tree)

| Характеристика | Описание |
|----------------|----------|
| Структура | Иерархическая структура из узлов и связей (родитель → потомки) |
| Основные элементы | Корень, узлы, листья |
| Особенности | Быстрый поиск, упорядочение и хранение данных |
| Использование | Файловые системы, DOM, базы данных, индексные структуры | 

Виды деревьев

| Вид дерева | Описание |
|------------|----------|
| Binary Tree | Каждый узел имеет до двух потомков |
| Binary Search Tree (BST) | Левый < родитель < правый, подходит для поиска |
| AVL / Red-Black Tree | Самобалансирующиеся деревья |
| Trie | Дерево для хранения строк по символам |
| Heap | Дерево для приоритетной очереди |

Сравнение структур

| Структура | Плюсы | Минусы |
|----------|-------|--------|
| Стек | Быстрый доступ к последнему элементу | Ограничен LIFO-логикой |
| Очередь | Сохраняет порядок поступления | Нет произвольного доступа |
| Дерево | Быстрый поиск и логическая организация данных | Сложнее в реализации |

Ключевые идеи

| Идея | Суть |
|------|------|
| LIFO | Используется там, где важен порядок вложенности |
| FIFO | Подходит для потоковой обработки данных |
| Иерархия | Деревья оптимальны для структурирования больших объёмов данных |

### Рефлексия и introspection <a id="рефлексия"></a>

Рефлексия и introspection — механизмы исследования и изменения объектов во время выполнения программы.

Introspection — это получение информации об объекте во время выполнения.

Основные понятия

| Понятие        | Описание |

Инструменты introspection

| Вызов функций           | Вызов методов по имени |

Области применения

| Область        | Зачем используется |
|---------------|-------------------|
| Фреймворки    | Автоматическое связывание компонентов |

Недостатки

| Минус | Описание |
|------|----------|
| Сложность | Код становится труднее для понимания |
| Ошибки времени выполнения | Ошибки проявляются позже |
| Производительность | Медленнее статического кода |
| Поддержка | Сложнее анализировать и рефакторить |

Ключевая идея
Introspection отвечает на вопрос «что это за объект?».
Рефлексия отвечает на вопрос «что с этим объектом можно сделать и изменить».

### Метаклассы <a id="метаклассы"></a>

Метаклассы в Python — это «классы для классов». Они управляют созданием и поведением классов.

Метакласс определяет, как создаётся класс, какие атрибуты и методы у него будут, и позволяет изменять класс на этапе его определения.

Назначение метаклассов

| Задача                       | Описание |
|-------------------------------|---------|
| Контроль создания классов     | Изменение или добавление атрибутов/методов |
| Автоматизация повторяющихся шаблонов | Генерация методов, проверка наследования |
| Валидация структуры классов   | Проверка обязательных атрибутов или методов |
| Расширение возможностей ООП   | Реализация паттернов на уровне классов |

Основные элементы

| Понятие       | Суть |
|---------------|------|
| type          | Базовый метакласс Python, создает все классы |
| __new__       | Метод, создающий класс перед его инициализацией |
| __init__      | Инициализация класса после его создания |
| __call__      | Управление вызовом класса (создание экземпляров) |

Создание метакласса

| Шаг                  | Описание | 
|---------------------|----------|
| Наследование от type | Метакласс создается как подкласс type |
| Переопределение __new__ | Добавление или изменение атрибутов класса |
| Переопределение __init__| Дополнительная логика после создания класса |
| Использование в классе  | class MyClass(metaclass=Meta) |

Применение метаклассов

| Сценарий                     | Зачем использовать | 
|-------------------------------|------------------|
| Регистрация классов           | Автоматическое хранение всех подклассов |
| Проверка интерфейсов          | Обеспечение наличия обязательных методов |
| Автоматическая генерация методов | Сокращение шаблонного кода |
| Singleton и другие паттерны   | Управление созданием экземпляров на уровне класса |

Важные свойства

| Свойство                  | Описание |
|---------------------------|----------|
| Метакласс = класс класса   | Класс сам является объектом type |
| Наследование               | Классы наследуют обычные атрибуты и поведение метакласса |
| Динамическое создание      | Можно создавать новые классы в runtime |
| Влияние на экземпляры      | Метакласс косвенно управляет поведением объектов класса |

Преимущества и риски

| Плюсы                       | Минусы |
|------------------------------|--------|
| Централизация логики создания | Сложность понимания кода |
| Автоматизация шаблонов       | Трудно отлаживать ошибки |
| Расширяемость ООП            | Используется редко, высокая абстракция |

Ключевая идея
Метаклассы позволяют вмешиваться в процесс создания классов, добавлять атрибуты и методы автоматически, контролировать наследование и реализовывать сложные паттерны на уровне классов.

### Динамическое создание классов <a id="динамические-классы"></a>

Динамическое создание классов в Python — это возможность создавать классы во время выполнения программы.

Python позволяет создавать классы не только через `class`, но и с помощью функции `type` или метаклассов.

Назначение динамических классов

| Задача                        | Описание |
|-------------------------------|----------|
| Генерация классов на лету     | Создание классов с уникальными атрибутами и методами |
| Автоматизация шаблонов        | Избегание повторяющегося кода |
| Плагины и расширяемость       | Подключение новых классов без изменения кода |
| ORM и фреймворки              | Динамическое создание моделей и сущностей |

Создание класса через `type`

| Шаг                  | Описание |
|---------------------|----------|
| Имя класса          | Строка с именем создаваемого класса | 
| Базовые классы       | Кортеж родительских классов |
| Атрибуты и методы    | Словарь с ключами — именами атрибутов и методов |

Пример структуры (без кода)

| Элемент             | Суть |
|--------------------|------|
| class_name          | Название нового класса |
| bases               | Наследуемые классы |
| attributes          | Методы, переменные класса |
| type(class_name, bases, attributes) | Создает новый класс динамически |

Применение

| Сценарий                          | Зачем использовать |
|----------------------------------|------------------|
| ORM                                 | Создание моделей таблиц на основе схем БД |
| API генераторы                      | Автоматическое создание ресурсов и методов |
| Плагины                             | Подключение новых классов без перекомпиляции |
| Декораторы и фабрики классов        | Изменение поведения классов на этапе выполнения |

Особенности

| Свойство                   | Описание | 
|----------------------------|----------|
| runtime creation            | Классы создаются во время выполнения |
| наследование                | Можно динамически указывать базовые классы |
| атрибуты и методы           | Легко добавлять новые функции и свойства |
| интеграция с метаклассами   | Можно комбинировать с метаклассами для расширенной логики |

Преимущества и риски

| Плюсы                         | Минусы |
|-------------------------------|--------|
| Гибкость                      | Сложнее читать и отлаживать |
| Автоматизация                 | Потенциально сложная архитектура |
| Поддержка динамических фреймворков | Редко встречается в простом коде |

Ключевая идея
Динамическое создание классов позволяет создавать и настраивать классы программно, обеспечивая гибкость и возможность генерации сложных структур на этапе выполнения.

## 🏗️ Архитектурные подходы и паттерны проектирования <a id="архитектура"></a>

### Принципы SOLID <a id="solid"></a>

**SOLID** — пять принципов объектно-ориентированного проектирования, которые делают код поддерживаемым, гибким и расширяемым.

#### Обзор принципов

| Принцип | Полное название | Ключевая идея |
|---------|-----------------|--------------|
| **S** | Single Responsibility | Один класс — одна ответственность |
| **O** | Open/Closed | Открыт для расширения, закрыт для изменения |
| **L** | Liskov Substitution | Объекты должны быть заменяемы |
| **I** | Interface Segregation | Много специализированных интерфейсов |
| **D** | Dependency Inversion | Зависимость от абстракций, а не реализаций |

#### Принцип единой ответственности (SRP)

| Аспект | Описание | Примеры в тестировании |
|--------|----------|------------------------|
| **Суть** | Класс должен иметь одну причину для изменения | Разделение тестов, утилит, конфигураций |
| **Нарушения** | Класс делает слишком много разных вещей | Тест проверяет UI, API и БД одновременно |
| **Польза** | Упрощение понимания и модификации | Легко находить и исправлять ошибки |
| **Применение** | Разделение логики тестов и вспомогательных функций | Отдельные классы для Page Objects, API клиентов, валидаторов |

#### Принцип открытости/закрытости (OCP)

| Аспект | Описание | Примеры в тестировании |
|--------|----------|------------------------|
| **Суть** | Класс открыт для расширения, но закрыт для изменений | Добавление новых тестов без изменения фреймворка |
| **Механизмы** | Наследование, композиция, интерфейсы | Базовые классы тестов, плагины pytest |
| **Нарушения** | Модификация существующего кода для добавления функциональности | Изменение базового класса при добавлении нового типа теста |
| **Польза** | Минимальный риск сломать существующую функциональность | Стабильность тестового фреймворка |

#### Принцип подстановки Барбары Лисков (LSP)

| Аспект | Описание | Примеры в тестировании |
|--------|----------|------------------------|
| **Суть** | Наследники должны быть взаимозаменяемы с родителями | Драйверы браузеров, разные реализации репортеров |
| **Требования** | Предусловия не усиливаются, постусловия не ослабляются | Интерфейс WebDriver для Chrome/Firefox |
| **Нарушения** | Переопределение методов с другими контрактами | Метод в наследнике бросает исключения, которых нет у родителя |
| **Польза** | Предсказуемость поведения инициализированных объектов | Возможность подменять реализации (моки, стабы) |

#### Принцип разделения интерфейса (ISP)

| Аспект | Описание | Примеры в тестировании |
|--------|----------|------------------------|
| **Суть** | Много специализированных интерфейсов лучше одного общего | Раздельные интерфейсы для UI тестов, API тестов, утилит |
| **Проблема** | Классы вынуждены реализовывать ненужные методы | Тестовый класс с методами для всех типов проверок |
| **Решение** | Разделение на мелкие ролевые интерфейсы | Интерфейсы `WebUITester`, `APITester`, `DatabaseValidator` |
| **Польза** | Избегание зависимостей от неиспользуемых методов | Чистая архитектура тестового фреймворка |

#### Принцип инверсии зависимостей (DIP)

| Аспект | Описание | Примеры в тестировании |
|--------|----------|------------------------|
| **Суть** | Зависимость от абстракций, а не конкретных реализаций | Фикстуры pytest, внедрение зависимостей |
| **Высокоуровневые модули** | Тесты бизнес-логики | Не зависят от конкретного драйвера браузера |
| **Низкоуровневые модули** | Реализации WebDriver | Реализуют интерфейсы для работы с браузером |
| **Польза** | Легкость подмены реализаций, тестируемость | Возможность использовать моки в unit-тестах |

#### Применение SOLID в тестовой автоматизации

##### Архитектура тестового фреймворка
| Принцип | Применение | Результат |
|---------|------------|-----------|
| **SRP** | Разделение page objects, тестов, утилит | Четкая организация кода |
| **OCP** | Расширение через наследование базовых классов | Легкое добавление новых типов тестов |
| **LSP** | Интерфейсы для драйверов, репортеров | Взаимозаменяемость компонентов |
| **ISP** | Специализированные интерфейсы для разных задач | Отсутствие "жирных" интерфейсов |
| **DIP** | Внедрение зависимостей через фикстуры | Гибкость и тестируемость |

##### Примеры нарушений и исправлений
| Нарушение | Проблема | SOLID решение |
|-----------|----------|---------------|
| **"Божественный объект" теста** | Один класс делает все | Разделение по SRP |
| **Модификация базовых классов** | Риск сломать все тесты | Расширение через OCP |
| **Жесткая связь с драйвером** | Невозможность подменить реализацию | DIP через интерфейсы |
| **"Толстый" интерфейс тестера** | Принудительная реализация ненужных методов | ISP на специализированные интерфейсы |

#### SOLID и паттерны проектирования 

| Паттерн | Какие принципы реализует | Применение в тестировании |
|---------|--------------------------|---------------------------|
| **Фабричный метод** | OCP, DIP | Создание драйверов браузеров |
| **Стратегия** | OCP, DIP, ISP | Разные стратегии ожидания |
| **Декоратор** | OCP, SRP | Добавление логирования к тестам |
| **Наблюдатель** | OCP, DIP | Система репортинга и нотификаций |
| **Адаптер** | LSP, ISP | Адаптация разных драйверов к общему интерфейсу | 

#### Влияние SOLID на качество тестов

| Качество кода | Как SOLID помогает | Пример |
|---------------|-------------------|--------|
| **Поддерживаемость** | Четкое разделение ответственности | Легко находить и исправлять ошибки |
| **Расширяемость** | Возможность добавлять функциональность без изменений | Новые типы тестов через наследование |
| **Тестируемость** | Зависимость от абстракций позволяет использовать моки | Unit-тесты для вспомогательных классов |
| **Читаемость** | Понятная структура и назначение классов | Интуитивно понятная архитектура |

#### Практические рекомендации

##### Для начинающих
| Шаг | Действие | SOLID принцип |
|-----|----------|---------------|
| **1** | Разделить тесты и вспомогательные функции | SRP |
| **2** | Использовать базовые классы для общих функций | OCP |
| **3** | Создавать интерфейсы для ключевых компонентов | ISP, DIP |
| **4** | Обеспечивать заменяемость реализаций | LSP |

##### Для опытных команд
| Практика | Реализация | Преимущество |
|----------|------------|--------------|
| **DI контейнер** | Внедрение зависимостей в фикстурах | Гибкость конфигурации |
| **Контрактное тестирование** | Интерфейсы с четкими контрактами | Предсказуемость |
| **Модульное тестирование фреймворка** | Тесты для вспомогательных классов | Качество инфраструктуры |

#### Инструменты и фреймворки, реализующие SOLID

| Инструмент | SOLID принципы | Пример реализации |
|------------|---------------|-------------------|
| **pytest** | DIP (фикстуры), OCP (плагины) | Система фикстур для DI |
| **Page Object Model** | SRP, ISP, DIP | Разделение страниц и тестов |
| **Allure/Reporting** | OCP, DIP | Плагинная архитектура репортеров |
| **Selenium WebDriver** | LSP, ISP | Интерфейсы для разных браузеров |

**Ключевой вывод:**
Принципы SOLID в тестовой автоматизации создают основу для создания масштабируемых, поддерживаемых и надежных тестовых фреймворков. Применение этих принципов позволяет строить архитектуру, которая легко адаптируется к изменениям и обеспечивает высокое качество тестового кода.

### Основные паттерны проектирования <a id="паттерны"></a>

**Паттерны проектирования** — проверенные решения повторяющихся проблем в проектировании программного обеспечения.

#### Классификация паттернов

| Категория | Назначение | Примеры |
|-----------|------------|---------|
| **Порождающие** | Создание объектов | Factory, Singleton, Builder |
| **Структурные** | Организация классов и объектов | Adapter, Decorator, Facade |
| **Поведенческие** | Взаимодействие между объектами | Strategy, Observer, Command |

#### Порождающие паттерны в тестировании

##### Фабричный метод (Factory Method)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Создание объектов через подклассы | Фабрика драйверов браузеров |
| **Проблема** | Жесткая привязка к конкретным классам | `new ChromeDriver()` в коде тестов |
| **Решение** | Интерфейс для создания, реализация в наследниках | `BrowserFactory.create_driver(browser_type)` |
| **Преимущества** | Гибкость, расширяемость, тестируемость | Легкое добавление новых браузеров |

##### Одиночка (Singleton)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Единственный экземпляр класса | Конфигурация тестового фреймворка |
| **Проблема** | Множественные экземпляры настроек | Противоречивые конфигурации |
| **Решение** | Глобальная точка доступа к единственному экземпляру | `Config.get_instance()` |
| **Риски** | Глобальное состояние, сложность тестирования | Использовать осторожно, предпочитать DI |

##### Строитель (Builder)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Пошаговое создание сложных объектов | Построение тестовых данных |
| **Проблема** | Сложные конструкторы с множеством параметров | `User(name, email, age, ...)` |
| **Решение** | Отдельный класс-строитель | `UserBuilder().with_name("John").with_email("...").build()` |
| **Преимущества** | Читаемость, гибкость, неизменяемость объектов | Fluent interface для создания тестовых объектов |

#### Структурные паттерны в тестировании

##### Адаптер (Adapter)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Преобразование интерфейса класса в другой интерфейс | Унификация разных драйверов |
| **Проблема** | Несовместимые интерфейсы | Selenium vs Playwright API |
| **Решение** | Класс-обертка, предоставляющий единый интерфейс | `WebDriverAdapter` для разных реализаций |
| **Преимущества** | Совместимость, повторное использование кода | Один набор тестов для разных инструментов |

##### Декоратор (Decorator) 
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Динамическое добавление функциональности | Логирование, тайминг, ретраи |
| **Проблема** | Наследование для добавления поведения | Множественные классы для комбинаций |
| **Решение** | Обертки, добавляющие поведение | `LoggingDriver(RetryDriver(base_driver))` |
| **Преимущества** | Гибкость, разделение ответственности | Композиция вместо наследования |

##### Фасад (Facade)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Упрощенный интерфейс к сложной системе | Упрощение работы с API/базами данных |
| **Проблема** | Сложность использования системы | Множество вызовов для одной операции |
| **Решение** | Единый класс с упрощенными методами | `DatabaseFacade` для CRUD операций |
| **Преимущества** | Упрощение клиентского кода, инкапсуляция | Чистые и понятные тесты |

#### Поведенческие паттерны в тестировании

##### Стратегия (Strategy)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Выбор алгоритма во время выполнения | Разные стратегии поиска элементов |
| **Проблема** | Множество условных операторов | `if headless: ... elif mobile: ...` |
| **Решение** | Интерфейс стратегии и конкретные реализации | `WaitStrategy`, `LocatorStrategy` |
| **Преимущества** | Гибкость, устранение условных операторов | Легкое добавление новых стратегий |

##### Наблюдатель (Observer)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Уведомление зависимых объектов об изменениях | Система репортинга и нотификаций |
| **Проблема** | Жесткая связь между компонентами | Тесты напрямую вызывают репортеры |
| **Решение** | Издатель и подписчики | `TestEventPublisher` и listeners |
| **Преимущества** | Слабая связность, расширяемость | Легкое добавление новых репортеров |

##### Команда (Command)
| Аспект | Описание | Пример в тестировании |
|--------|----------|------------------------|
| **Суть** | Инкапсуляция запроса как объекта | История действий, отмена операций |
| **Проблема** | Прямой вызов методов без возможности управления | Невозможность логирования/повтора операций |
| **Решение** | Объекты-команды с методом execute | `ClickCommand`, `TypeCommand` |
| **Преимущества** | Отслеживание, отмена, повтор операций | Скриншоты при ошибках, повторное выполнение |

#### Применение паттернов в тестовых фреймворках

##### Page Object Model как паттерн
| Компонент | Реализуемые паттерны | Преимущества |
|-----------|----------------------|--------------|
| **Page Objects** | Facade (для страницы), Builder (для данных) | Инкапсуляция деталей страницы | 
| **Page Factory** | Factory Method, Abstract Factory | Создание page objects |
| **Page Elements** | Composite, Proxy | Иерархия элементов |

##### Архитектура тестового фреймворка
| Слой | Используемые паттерны | Цель |
|------|-----------------------|------|
| **Тестовый слой** | Template Method, Strategy | Организация тестов |
| **Бизнес-слой** | Facade, Command | Высокоуровневые действия |
| **Инфраструктурный слой** | Adapter, Factory, Singleton | Работа с внешними системами |
| **Утилитарный слой** | Builder, Decorator | Вспомогательные функции |

#### Антипаттерны и как их избежать

| Антипаттерн | Проблема | Паттерн-решение |
|-------------|----------|-----------------|
| **God Object** | Один класс делает все | SRP + Facade + Strategy |
| **Spaghetti Code** | Запутанные зависимости | Observer + Dependency Injection |
| **Hard Code** | Жесткие зависимости | Factory + Adapter |
| **Copy-Paste** | Дублирование кода | Template Method + Strategy |

#### Критерии выбора паттернов

| Критерий | Подходящие паттерны | Пример ситуации |
|----------|---------------------|-----------------|
| **Создание объектов** | Factory, Builder, Singleton | Драйверы, тестовые данные |
| **Расширение функциональности** | Decorator, Strategy | Логирование, разные алгоритмы |

| **Управление зависимостями** | Observer, Dependency Injection | Система событий, конфигурация |

**Ключевой вывод:**
Паттерны проектирования в тестовой автоматизации решают стандартные проблемы архитектуры, делая код более поддерживаемым, гибким и расширяемым. Правильный выбор паттернов зависит от конкретных задач: создание объектов, организация структуры или управление поведением системы.

### Dependency Injection и IoC <a id="dependency-injection"></a>

**Dependency Injection (DI) и Inversion of Control (IoC)** — принципы проектирования, которые делают код более гибким, тестируемым и поддерживаемым.

#### Основные понятия

| Термин | Определение | Аналогия |
|--------|-------------|----------|
| **Inversion of Control (IoC)** | Перенос управления зависимостями из класса во внешний компонент | Ресторан: вы не готовите, а заказываете еду |
| **Dependency Injection (DI)** | Техника реализации IoC путем внедрения зависимостей извне | Доставка ингредиентов на кухню |
| **Dependency** | Объект, который требуется классу для работы | Ингредиенты для приготовления блюда |
| **Container** | Компонент, управляющий зависимостями и их жизненным циклом | Шеф-повар, который решает что и когда готовить |

#### Принципы и преимущества

| Принцип | Суть | Преимущество в тестировании |
|---------|------|-----------------------------|
| **Инверсия зависимостей** | Классы зависят от абстракций, а не конкретных реализаций | Легкая подмена реальных компонентов моками |
| **Разделение ответственности** | Создание объектов отделено от их использования | Чистая архитектура, каждый класс делает одно |
| **Конфигурируемость** | Зависимости настраиваются извне | Разные конфигурации для dev/test/prod |
| **Тестируемость** | Изоляция компонентов для unit-тестирования | Mocking и stubbing без сложных хаков |

#### Типы Dependency Injection

##### Constructor Injection
| Аспект | Описание | Когда использовать |
|--------|----------|-------------------|
| **Суть** | Зависимости передаются через конструктор | Обязательные зависимости |
| **Преимущества** | Явные зависимости, неизменяемость | Всегда предпочтительный метод |
| **Недостатки** | Много параметров в конструкторе | Для большого количества зависимостей |
| **Пример** | `class TestRunner(driver, reporter, config)` | Базовые компоненты тестового фреймворка |

##### Setter Injection
| Аспект | Описание | Когда использовать |
|--------|----------|-------------------|
| **Суть** | Зависимости устанавливаются через setter-методы | Опциональные зависимости |
| **Преимущества** | Гибкость, возможность изменения зависимостей | Динамическая конфигурация |
| **Недостатки** | Скрытые зависимости, mutable состояние | Когда зависимости могут меняться |
| **Пример** | `test.set_driver(driver)` | Плагины, расширения |

##### Interface Injection
| Аспект | Описание | Когда использовать |
|--------|----------|-------------------|
| **Суть** | Зависимость предоставляет интерфейс для инъекции | Специализированные сценарии |
| **Преимущества** | Четкие контракты, типобезопасность | Когда нужен строгий контроль |
| **Недостатки** | Сложность реализации | Редко используется в Python |
| **Пример** | Интерфейс `Injectable` с методом `inject()` | Сложные фреймворки |

#### IoC Container / DI Container

##### Функции контейнера
| Функция | Описание | Пример в тестировании |
|---------|----------|------------------------|
| **Регистрация** | Связывание интерфейсов с реализациями | `container.register(WebDriver, ChromeDriver)` |
| **Разрешение** | Создание объектов с внедренными зависимостями | `runner = container.resolve(TestRunner)` |
| **Управление жизненным циклом** | Singleton, Transient, Scoped | Драйвер на тест, конфигурация на сессию |
| **Валидация** | Проверка корректности зависимостей | Обнаружение циклических зависимостей |

##### Популярные DI контейнеры для Python
| Библиотека | Особенности | Рекомендация |
|------------|-------------|--------------|
| **dependency-injector** | Полнофункциональный, типизированный | Для сложных проектов |
| **injector** | Простой, легковесный | Для средних проектов |
| **punq** | Минималистичный, быстрый | Для небольших проектов |
| **Кастомный контейнер** | Полный контроль, простота | Для обучения или специфичных нужд |

#### Применение в тестовой автоматизации

##### Архитектура с DI
| Компонент | Роль в DI | Пример зависимости |
|-----------|-----------|-------------------|
| **Тестовый фреймворк** | IoC Container | Управляет всеми зависимостями |
| **Тестовые классы** | Потребители зависимостей | Принимают драйвер, репортер, конфиг |
| **Сервисы** | Предоставляемые зависимости | WebDriver, APIClient, Database |
| **Конфигурация** | Источник настроек | Environment variables, config files |

##### Пример DI в тестовом фреймворке
```python
class TestRunner:
    def __init__(self, driver: WebDriver, reporter: Reporter, config: Config):
        self.driver = driver  # Constructor Injection
        self.reporter = reporter
        self.config = config
    
    def run_test(self, test_case):
        # Использование внедренных зависимостей
        self.driver.get(self.config.base_url)
        # ... выполнение теста
        self.reporter.report_result(test_case.result)

# Контейнер регистрирует зависимости
container = Container()
container.register(WebDriver, ChromeDriver)
container.register(Reporter, AllureReporter)
container.register(Config, TestConfig)

# Контейнер создает TestRunner со всеми зависимостями
runner = container.resolve(TestRunner)
```

#### DI и фикстуры pytest

##### Сравнение подходов
| Подход | DI контейнер | pytest фикстуры |
|--------|--------------|-----------------|
| **Централизация** | Единый контейнер управления | Разрозненные фикстуры в conftest.py |
| **Типизация** | Строгая типизация (Type hints) | Динамическая типизация |
| **Конфигурируемость** | Конфигурационные файлы, код | Параметры командной строки, маркеры |
| **Сложность** | Высокая начальная настройка | Низкий порог входа |
| **Гибкость** | Структурированная, предсказуемая | Быстрая, гибкая |

##### Интеграция DI с pytest 
| Метод | Реализация | Преимущества |
|--------|------------|--------------|
| **DI через фикстуры** | Фикстуры как DI контейнер | Использование существующей инфраструктуры |
| **Кастомные маркеры** | Маркеры для выбора реализаций | Гибкость в выборе зависимостей |
| **Плагины pytest** | Плагины для DI интеграции | Стандартизированный подход |

#### Паттерны и лучшие практики

##### Composition Root
| Концепция | Описание | Реализация |
|-----------|----------|------------|
| **Суть** | Единое место в приложении, где составляется граф объектов | Инициализация в `conftest.py` или главном модуле |
| **Преимущества** | Контроль над созданием объектов, централизация | Легкость отладки и модификации |
| **Пример** | Создание всех зависимостей перед запуском тестов | Настройка контейнера в pytest_configure |

##### Constructor Injection как стандарт
| Правило | Причина | Исключения |
|---------|---------|------------|
| **Всегда использовать constructor injection** | Явность, неизменяемость, тестируемость | Опциональные зависимости через setter |
| **Избегать service locator** | Скрытые зависимости, сложность тестирования | Legacy код, постепенная миграция |
| **Интерфейсы вместо классов** | Гибкость, подмена реализаций | Когда абстракция излишня |

#### Примеры использования в тестировании

##### Тестирование с моками
```python

        self.http_client = http_client  # DI для тестирования

# В production

# В тестах

```

```python

        self.connection_string = config.connection_string

# Разные конфигурации для разных окружений
test_config = DatabaseConfig("sqlite://:memory:")
```

#### Антипаттерны и ошибки

| Антипаттерн | Проблема | Решение |
|-------------|----------|---------|
| **Service Locator** | Скрытые зависимости, сложность тестирования | Использовать Constructor Injection |
| **God Container** | Контейнер знает слишком много, становится точкой отказа | Разделение ответственности, модульные контейнеры |
| **Circular Dependencies** | Циклические зависимости, сложность разрешения | Рефакторинг, введение интерфейсов |
| **Over-engineering** | Слишком сложная DI система для простого проекта | Начинать с простых фикстур, добавлять DI при росте |

#### Миграция на DI

##### Пошаговый подход
| Шаг | Действие | Пример |
|-----|----------|--------|
| **1** | Выделение зависимостей в конструкторы | `def __init__(self, driver):` |
| **2** | Создание интерфейсов/абстрактных классов | `class WebDriver(ABC):` |
| **3** | Внедрение простого контейнера | Фабрика или словарь зависимостей |
| **4** | Интеграция с DI библиотекой | `dependency-injector` или `injector` |

**Ключевой вывод:**
Dependency Injection и Inversion of Control — мощные инструменты для создания гибких, тестируемых и поддерживаемых тестовых фреймворков. Хотя начальная настройка требует усилий, долгосрочные выгоды в виде чистого кода, легкого тестирования и гибкой архитектуры полностью оправдывают инвестиции.

### Архитектурные стили (MVC, MVT, Clean Architecture) <a id="архитектурные-стили"></a>

**Архитектурные стили** — проверенные шаблоны организации кода, которые определяют взаимодействие компонентов системы.

#### Сравнительный обзор

| Критерий | MVC (Model-View-Controller) | MVT (Model-View-Template) | Clean Architecture |
|----------|-----------------------------|---------------------------|-------------------|
| **Происхождение** | Smalltalk-80 (1979) | Django (2005) | Robert C. Martin (2012) |
| **Основная идея** | Разделение данных, представления и логики | Автоматическая связь модели и представления | Независимость от фреймворков и внешних зависимостей |
| **Ключевые компоненты** | Model, View, Controller | Model, View, Template | Entities, Use Cases, Controllers, Gateways |
| **Связность** | Средняя | Высокая (Django-specific) | Низкая (максимальная гибкость) |

#### MVC (Model-View-Controller)

##### Компоненты
| Компонент | Роль | Ответственность |
|-----------|------|-----------------|
| **Model** | Данные и бизнес-логика | Хранение состояния, валидация, бизнес-правила |
| **View** | Представление | Отображение данных пользователю |
| **Controller** | Посредник | Обработка ввода, обновление Model и View |

##### Поток данных
```
Пользователь → Controller → Model → View → Пользователь
    ↓           ↓           ↓        ↓
   Ввод      Обработка   Обновление  Отображение
            запроса      данных      результата
```

##### Плюсы и минусы
| Преимущества | Недостатки | Когда использовать |
|--------------|------------|-------------------|
| **Четкое разделение ответственности** | Усложнение для простых проектов | Веб-приложения, десктоп приложения |
| **Переиспользование компонентов** | Избыточность кода в Controller | Проекты с разными представлениями |
| **Параллельная разработка** | Сложность отладки цепочки вызовов | Командная разработка |
| **Гибкость представлений** | Зависимость View от Model | Мультиплатформенные приложения |

#### MVT (Model-View-Template) - Django подход

##### Компоненты
| Компонент | Роль в Django | Аналог в MVC |
|-----------|--------------|--------------|
| **Model** | Определение структуры данных и БД логики | Model |
| **View** | Контроллер + бизнес-логика | Controller |
| **Template** | Представление (HTML с Django Template Language) | View |

##### Особенности Django MVT
| Особенность | Описание | Преимущество |
|-------------|----------|--------------|
| **Автоматическая админка** | Генерируется из Model | Быстрая разработка CRUD |
| **URL dispatcher** | Маршрутизация через urls.py | Чистое разделение маршрутов |
| **ORM в ядре** | Интегрированный доступ к БД | Единообразие работы с данными |
| **Батарейки включены** | Готовые компоненты (auth, admin, forms) | Скорость разработки |

##### Поток запроса в Django
```
Запрос → URL Router → View → Model → Template → Ответ
    ↓        ↓         ↓        ↓        ↓        ↓
   HTTP   Определение  Обработка  Работа  Генерация  HTTP
          обработчика  логики     с БД    HTML      ответ
```

#### Clean Architecture (Чистая Архитектура)

##### Концентрические круги
| Уровень | Назначение | Зависимости |
|---------|------------|-------------|
| **Entities** | Бизнес-правила и объекты | Нет внешних зависимостей |
| **Use Cases** | Специфичная бизнес-логика | Только Entities |
| **Interface Adapters** | Конвертеры данных | Use Cases, External |
| **Frameworks & Drivers** | Внешние системы и фреймворки | Все внутренние слои |

##### Принципы
| Принцип | Суть | Практическое применение |
|---------|------|-------------------------|
| **Dependency Rule** | Зависимости направлены внутрь | Внешние слои зависят от внутренних |
| **Abstraction** | Зависимость от абстракций | Интерфейсы вместо конкретных классов |
| **Separation of Concerns** | Разделение ответственности | Каждый слой имеет свою зону ответственности |

##### Компоненты Clean Architecture
| Компонент | Расположение | Пример |
|-----------|--------------|--------|
| **Entities** | Центр | User, Product, Order |
| **Use Cases / Interactors** | Второй круг | CreateUser, ProcessOrder |
| **Controllers / Presenters** | Третий круг | REST контроллеры, GraphQL resolvers |
| **Gateways / Repositories** | Граница | DatabaseGateway, ExternalAPIGateway |
| **Frameworks** | Внешний круг | Django, Flask, SQLAlchemy |

#### Применение в тестовой автоматизации

##### Тестирование MVC приложений
| Компонент | Стратегия тестирования | Инструменты |
|-----------|------------------------|-------------|
| **Model** | Unit тесты бизнес-логики | pytest, unittest |
| **View** | UI тесты, скриншотные тесты | Selenium, Playwright |
| **Controller** | API тесты, интеграционные тесты | requests, pytest |

##### Тестирование Django (MVT) приложений
| Компонент | Подход Django тестирования | Особенности |
|-----------|---------------------------|-------------|
| **Models** | TestCase с тестовой БД | `django.test.TestCase` |
| **Views** | RequestFactory или тестовый клиент | `client.get()`, `client.post()` |
| **Templates** | Проверка контекста и рендеринга | `assertTemplateUsed()` |
| **Forms** | Валидация данных формы | `form.is_valid()` |

##### Тестирование Clean Architecture
| Слой | Тип тестов | Mocking стратегия |
|------|------------|-------------------|
| **Entities** | Unit тесты | Не требуют моков |
| **Use Cases** | Unit тесты | Mock репозитории и гейтвеи |
| **Interface Adapters** | Интеграционные тесты | Фейковые внешние системы |
| **Frameworks** | E2E тесты | Реальные системы |

#### Миграция между архитектурами

##### От MVC/MVT к Clean Architecture
| Этап | Действие | Сложность |
|------|----------|-----------|
| **Выделение Entities** | Извлечение бизнес-объектов из моделей | Низкая |
| **Создание Use Cases** | Перенос бизнес-логики из контроллеров/вьюх | Средняя |
| **Введение репозиториев** | Абстракция доступа к данным | Высокая |
| **Рефакторинг зависимостей** | Следование Dependency Rule | Очень высокая |

##### Гибридные подходы
| Комбинация | Преимущества | Недостатки |
|------------|--------------|------------|
| **Django + Clean Architecture** | Быстрая разработка + чистая архитектура | Когнитивная нагрузка |
| **Flask/FastAPI + MVC** | Легковесность + структурированность | Меньше встроенных возможностей |
| **Современный MVC** | MVC с элементами Clean Architecture | Баланс простоты и гибкости |

#### Выбор архитектуры для проекта

##### Критерии выбора
| Критерий | MVC | MVT (Django) | Clean Architecture |
|----------|-----|--------------|-------------------|
| **Скорость разработки** | Высокая | Очень высокая | Низкая (начальные затраты) |
| **Поддерживаемость** | Средняя | Средняя | Очень высокая |
| **Гибкость** | Высокая | Средняя (Django-way) | Очень высокая |
| **Кривая обучения** | Низкая | Средняя | Высокая |
| **Тестируемость** | Средняя | Высокая (встроенная) | Очень высокая |

##### Рекомендации по применению
| Тип проекта | Рекомендуемая архитектура | Причина |
|-------------|---------------------------|---------|
| **Прототип, MVP** | MVT (Django) или MVC | Быстрое время выхода на рынок |
| **Корпоративное приложение** | Clean Architecture | Долгосрочная поддерживаемость |
| **Микросервисы** | Clean Architecture или MVC | Гибкость, независимость сервисов |
| **Учебный проект** | MVC | Простота понимания |

#### Влияние на тестовую стратегию

##### Архитектура определяет тестирование
| Архитектура | Фокус тестирования | Типы тестов |
|-------------|-------------------|-------------|
| **MVC** | Интеграция компонентов | UI, API, интеграционные |
| **MVT** | Работа с фреймворком | Django-specific тесты |
| **Clean Architecture** | Изоляция слоев | Unit, интеграционные, контрактные |

##### Пример тестовой пирамиды
```
Clean Architecture:      MVC/MVT:
      [E2E]                [UI]
     /     \               /  \
[Интеграционные]     [API/Интеграционные]
         |                    |
     [Unit]               [Unit]
```

**Ключевой вывод:**
Выбор архитектурного стиля определяет не только структуру приложения, но и подход к его тестированию. Clean Architecture предлагает максимальную тестируемость и гибкость, но требует больше начальных усилий. MVC и MVT обеспечивают более быстрый старт, но могут создавать проблемы с поддержкой в долгосрочной перспективе.

```python
import os
import sys
from datetime import datetime

import requests
import pandas as pd

from .utils import helper_function


class DataProcessor:
    """Класс для обработки данных."""
    
    DEFAULT_TIMEOUT = 30
    
    def __init__(self, source_url, cache_dir=None):
        self.source_url = source_url
        self.cache_dir = cache_dir or "/tmp/cache"
        self._data = None
    
    def fetch_data(self, timeout=None):
        """Загрузка данных из источника."""
        if timeout is None:
            timeout = self.DEFAULT_TIMEOUT
        
        try:
            response = requests.get(
                self.source_url,
                timeout=timeout
            )
            response.raise_for_status()
            return response.json()
        except requests.RequestException as e:
            print(f"Ошибка загрузки данных: {e}")
            return None
    
    def process_data(self, data):
        """Обработка загруженных данных."""
        if not data:
            return []
        
        return [
            self._transform_item(item)
            for item in data
            if self._is_valid(item)
        ]
    
    def _transform_item(self, item):
        """Внутренний метод преобразования элемента."""
        return {
            'id': item.get('id'),
            'name': item.get('name', '').strip(),
            'timestamp': datetime.now()
        }
    
    def _is_valid(self, item):
        """Проверка валидности элемента."""
        return bool(item.get('id')) and bool(item.get('name'))
```

### Области видимости (LEGB) <a id="legb"></a>

**LEGB** — правило поиска переменных в Python, определяющее порядок поиска имени в разных областях видимости.

#### Иерархия областей видимости

| Уровень | Расшифровка | Описание | Время жизни |
|---------|-------------|----------|-------------|
| **L** | Local (локальная) | Внутри функции или класса | Время выполнения функции |
| **E** | Enclosing (охватывающая) | В объемлющих функциях (замыкания) | Пока существует объемлющая функция |
| **G** | Global (глобальная) | На уровне модуля | Время выполнения программы |
| **B** | Built-in (встроенная) | Встроенные имена Python | Всегда доступны |

#### Детальное описание каждой области 

##### Local (Локальная область)
| Характеристика | Описание | Пример |
|----------------|----------|--------|
| **Область действия** | Внутри функции или метода | Параметры, локальные переменные |
| **Время жизни** | Создается при вызове, уничтожается при возврате | Живет во время выполнения функции |
| **Доступность** | Только внутри функции | Не доступна извне |
| **Ключевое слово** | `local` (явное объявление) | Редко используется |

```python
def calculate_sum(a, b):
    # a, b, result - локальные переменные
    result = a + b
    return result

# Доступ только внутри функции
# print(result)  # NameError: name 'result' is not defined
```

##### Enclosing (Охватывающая область)
| Характеристика | Описание | Особенности |
|----------------|----------|-------------|
| **Контекст** | Вложенные функции (замыкания) | Nonlocal переменные |
| **Поиск** | От внутренней к внешней функции | Рекурсивный поиск вверх |
| **Модификация** | Ключевое слово `nonlocal` | Для изменения внешних переменных |
| **Использование** | Фабрики функций, декораторы | Сохранение состояния |

```python
def outer_function():
    # enclosing область для inner_function 
    outer_var = "I'm from outer function"
    
    def inner_function():
        # Имеет доступ к outer_var
        print(f"Inner sees: {outer_var}")
    
    inner_function()
    return inner_function

# Замыкание
def make_multiplier(n):
    def multiplier(x):
        # n из enclosing области
        return x * n
    return multiplier

double = make_multiplier(2)
triple = make_multiplier(3)
print(double(5))  # 10
print(triple(5))  # 15
```

##### Global (Глобальная область)
| Характеристика | Описание | Особенности |
|----------------|----------|-------------|
| **Область действия** | Весь модуль (.py файл) | Видна во всех функциях модуля |
| **Объявление** | На верхнем уровне модуля | Вне функций и классов |
| **Модификация** | Ключевое слово `global` | Для изменения внутри функций |
| **Хорошая практика** | Константы, конфигурация | Избегать изменяемых глобальных переменных |

```python
# Глобальные переменные
GLOBAL_CONSTANT = 3.14
config = {"debug": True}

def use_global():
    # Чтение глобальной переменной (без global)
    print(f"Constant: {GLOBAL_CONSTANT}")
    
    # Изменение требует global
    global config
    config["debug"] = False

def bad_practice():
    # Создает локальную переменную, а не изменяет глобальную
    GLOBAL_CONSTANT = 2.71  # Это НЕ изменит глобальную переменную
```

##### Built-in (Встроенная область)
| Характеристика | Описание | Примеры |
|----------------|----------|---------|
| **Содержимое** | Встроенные функции и типы | `print()`, `len()`, `str`, `list` |
| **Доступность** | Всегда доступны | Не требуют импорта |
| **Переопределение** | Возможно (но опасно) | Может сломать код |
| **Просмотр** | `dir(__builtins__)` | Список всех встроенных имен |

```python
# Встроенные функции доступны всегда
```

#### Правила поиска переменных

##### Алгоритм LEGB
```
Поиск переменной x:
1. Локальная область (внутри текущей функции)
2. Охватывающая область (внешние функции, если есть)
3. Глобальная область (уровень модуля)
4. Встроенная область (built-in имена)
```

##### Пример поиска
```python
x = "global x"  # Глобальная

def outer():
    x = "outer x"  # Enclosing для inner
    
    def inner():
        x = "inner x"  # Локальная для inner
        print(f"Local: {x}")  # "inner x"
    
    inner()
    print(f"Enclosing: {x}")  # "outer x"

outer()
print(f"Global: {x}")  # "global x"
```

#### Ключевые слова для работы с областями видимости

##### `global`
| Назначение | Синтаксис | Когда использовать |
|------------|-----------|-------------------|
| **Объявление глобальной переменной** | `global var_name` | Изменение глобальных переменных в функциях |
| **Создание глобальной переменной** | `global new_var; new_var = value` | Создание глобальных переменных из функций |

```python
counter = 0

def increment():
    global counter  # Указываем, что используем глобальную переменную
    counter += 1

def create_global():
    global NEW_GLOBAL  # Создаем новую глобальную переменную
    NEW_GLOBAL = "Created in function"

increment()
print(counter)  # 1

create_global()
print(NEW_GLOBAL)  # "Created in function"
```

##### `nonlocal`
| Назначение | Синтаксис | Когда использовать |
|------------|-----------|-------------------|
| **Работа с enclosing переменными** | `nonlocal var_name` | Изменение переменных из внешней функции |
| **Только для nested функций** | Работает только с enclosing областью | Не работает с глобальными переменными |

```python
def outer():
    count = 0
    
    def increment():
        nonlocal count  # Указываем, что count из enclosing области
        count += 1
        return count
    
    return increment

counter = outer()
print(counter())  # 1
print(counter())  # 2
print(counter())  # 3
```

#### Особые случаи и нюансы

##### Затенение переменных (Shadowing)
| Ситуация | Пример | Результат |
|----------|--------|-----------|
| **Локальная затеняет глобальную** | Глобальная `x`, локальная `x` | Внутри функции используется локальная |
| **Enclosing затеняет глобальную** | Глобальная `x`, enclosing `x` | Во внутренней функции используется enclosing |
| **Built-in затеняется** | `list = [1, 2, 3]` | Переопределение встроенной функции |

```python
list = "I'm not a list anymore"  # Опасно! 
# Теперь list() не работает как встроенная функция
```

##### Области видимости в классах
| Область | Доступность | Особенности |
|---------|-------------|-------------|
| **Атрибуты экземпляра** | `self.attribute` | Уникальны для каждого экземпляра |
| **Атрибуты класса** | `ClassName.attribute` | Общие для всех экземпляров |
| **Локальные методы** | Внутри методов | Как обычные локальные переменные |

```python
class MyClass:
    class_var = "I'm class variable"  # Атрибут класса
    
    def __init__(self, instance_var):
        self.instance_var = instance_var  # Атрибут экземпляра
    
    def method(self):
        local_var = "I'm local"  # Локальная переменная метода
        print(self.instance_var)  # Доступ к атрибуту экземпляра
        print(self.class_var)     # Доступ к атрибуту класса через self
        print(MyClass.class_var)  # Доступ к атрибуту класса через класс
```

##### Области видимости в comprehensions и генераторах
| Конструкция | Область видимости | Особенности |
|-------------|-------------------|-------------|
| **List/dict/set comprehension** | Изолированная | Не утекают переменные наружу |
| **Generator expression** | Изолированная | Аналогично comprehensions |
| **Циклы for/while** | Текущая область | Переменные цикла доступны после |

```python
# Comprehension создает свою область видимости
x = 10
result = [x for x in range(5)]  # Не изменяет внешнюю x
print(x)  # 10 (осталась неизменной)
print(result)  # [0, 1, 2, 3, 4]

# Цикл for использует текущую область
for x in range(5):
    pass
print(x)  # 4 (последнее значение)
```

```python
def make_counter(start=0):
    count = start  # Переменная в enclosing области
    
    def counter():
        nonlocal count
        current = count
        count += 1
        return current
    
    return counter

# Создаем независимые счетчики
```

###### Декораторы с состоянием
```python
def call_counter(func):
    count = 0  # Состояние в enclosing области
    
    def wrapper(*args, **kwargs):
        nonlocal count
        count += 1
        print(f"Function {func.__name__} called {count} times")
        return func(*args, **kwargs)
    
    return wrapper

@call_counter
def greet(name):
    return f"Hello, {name}!"

greet("Alice")  # Function greet called 1 times
greet("Bob")    # Function greet called 2 times
```

#### Распространенные ошибки

##### Ошибки с областями видимости
| Ошибка | Пример | Причина | Решение |
|--------|--------|---------|---------|
| **UnboundLocalError** | `x += 1` когда x глобальная | Python считает x локальной | Использовать `global x` |
| **NameError** | Использование до объявления | Переменная не найдена ни в одной области | Проверить объявление |
| **Неожиданное поведение** | Изменение изменяемого глобального объекта | Ссылочные типы изменяются без global | Явное копирование или global |

```python
# UnboundLocalError
x = 10

def problematic():
    print(x)  # Это работает (чтение)
    x = 20    # UnboundLocalError: local variable 'x' referenced before assignment
    # Python видит x = 20 и считает x локальной для всей функции

def solution():
    global x  # Объявляем x как глобальную
    print(x)  # 10
    x = 20    # Теперь работает
```

#### Best Practices

##### Рекомендации по использованию областей видимости
| Практика | Рекомендация | Обоснование |
|----------|--------------|-------------|
| **Избегать глобальных переменных** | Использовать параметры и возвращаемые значения | Предсказуемость, тестируемость |
| **Явные объявления** | Всегда использовать `global`/`nonlocal` при необходимости | Читаемость кода |
| **Константы как глобальные** | Глобальные только для неизменяемых констант | Безопасность, ясность |
| **Локальность** | Объявлять переменные как можно ближе к использованию | Упрощение понимания |

##### Антипаттерны
| Антипаттерн | Проблема | Альтернатива |
|-------------|----------|--------------|
| **Изменяемые глобальные переменные** | Непредсказуемое состояние | Классы, замыкания, DI | 
| **Слишком глубокое вложение** | Сложность отслеживания областей | Рефакторинг, разделение функций |
| **Неявные зависимости** | Скрытые использования глобальных переменных | Явная передача параметров |

```python
class FileLogger:
    """Кастомный контекстный менеджер для логирования в файл"""
    
    def __init__(self, filename, mode='a'):
        self.filename = filename
        self.mode = mode
        self.file = None
    
    def __enter__(self):
        """Вызывается при входе в контекст"""
        self.file = open(self.filename, self.mode)
        self.file.write(f"=== Начало сессии: {datetime.now()} ===\n")
        return self.file
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        """Вызывается при выходе из контекста"""
        if exc_type:
            self.file.write(f"Ошибка: {exc_type.__name__}: {exc_val}\n")
        
        self.file.write(f"=== Конец сессии: {datetime.now()} ===\n\n")
        self.file.close()
        
        # Возврат True подавляет исключение
        return False  # Пропускаем исключение дальше

# Использование
with FileLogger('app.log') as log_file:
    log_file.write("Выполнение операции\n")
    # Файл автоматически закрывается
```

```python
def timer_context(name="Операция"):
```

##### Транзакционные операции
```python
class DatabaseTransaction:
    """Контекстный менеджер для транзакций БД"""
    
    def __init__(self, connection):
        self.connection = connection
    
    def __enter__(self):
        self.connection.begin()
        return self.connection
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        if exc_type:
            self.connection.rollback()
            print("Транзакция откачена")
        else:
            self.connection.commit()
            print("Транзакция завершена успешно")

# Использование
with DatabaseTransaction(db_connection) as conn:
    conn.execute("INSERT INTO users VALUES (?, ?)", ("Alice", 25))
    conn.execute("UPDATE stats SET count = count + 1")
    # При ошибке - автоматический rollback
```

```python
class TestLoggingContext:
    """Контекст для логирования выполнения тестов"""
    
    def __init__(self, test_name):
        self.test_name = test_name
        self.log_file = None
    
    def __enter__(self):
        self.log_file = open(f'logs/{self.test_name}.log', 'w')
        self.log_file.write(f"Тест: {self.test_name}\n")
        self.log_file.write(f"Время начала: {datetime.now()}\n")
        self.log_file.write("=" * 50 + "\n")
        return self
    
    def log_step(self, step_name):
        self.log_file.write(f"Шаг: {step_name}\n")
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        if exc_type:
            self.log_file.write(f"ОШИБКА: {exc_type.__name__}: {exc_val}\n")
        else:
            self.log_file.write("ТЕСТ ПРОЙДЕН УСПЕШНО\n")
        
        self.log_file.write(f"Время завершения: {datetime.now()}\n")
        self.log_file.close()

# Использование в тестах
def test_feature():
    with TestLoggingContext("test_feature") as logger:
        logger.log_step("Настройка")
        setup()
        
        logger.log_step("Выполнение")
        result = operation()
```

## 🚀 Асинхронность и многопоточность <a id="асинхронность"></a>

### AsyncIO, Await и Event Loop <a id="asyncio"></a>

AsyncIO — стандартная библиотека Python для асинхронного и неблокирующего выполнения задач
в одном потоке с использованием цикла событий.

Она предназначена для эффективной работы с I/O-bound операциями.

Базовые понятия

| Понятие        | Описание |
|----------------|----------|
| Event Loop     | Центральный механизм управления задачами |
| Coroutine      | Асинхронная функция |
| Task           | Обёртка корутины для планирования |
| Future         | Объект результата, доступного позже |
| Awaitable      | Объект, который можно ожидать |

Event Loop

Event Loop управляет выполнением корутин и переключает их
в моменты ожидания I/O-операций.

| Функция Event Loop | Назначение |
|--------------------|------------|
| Планирование       | Запуск и переключение задач |
| Ожидание I/O       | Работа с сокетами и таймерами |
| Управление задачами| Контроль жизненного цикла |
| Обработка событий  | Реакция на завершение операций |

Coroutine и await

Coroutine — это функция, выполнение которой может быть приостановлено и продолжено позже.

| Элемент | Роль |
|--------|------|
| async def | Определяет корутину |
| await    | Приостанавливает выполнение |
| return   | Завершает корутину |
| yield    | Используется в async-генераторах |

Модель выполнения

| Шаг | Описание |
|-----|----------|
| Регистрация | Корутины добавляются в Event Loop |
| Запуск | Event Loop начинает выполнение |
| Ожидание | Задачи приостанавливаются на I/O |
| Переключение | Выполняются другие задачи |
| Завершение | Результат возвращается |

Сравнение с другими моделями

| Модель           | Подходит для | Особенности |
|------------------|--------------|-------------|
| asyncio          | I/O-bound    | Один поток, Event Loop |
| threading        | I/O-bound    | GIL, общая память |
| multiprocessing  | CPU-bound    | Несколько процессов |

Преимущества

| Плюс                    | Описание |
|-------------------------|----------|
| Высокая масштабируемость| Тысячи соединений |
| Низкие накладные расходы| Один поток |
| Управляемый контроль    | Явные точки ожидания |
| Предсказуемость         | Нет гонок данных |

Ограничения

| Минус                | Описание |
|----------------------|----------|
| CPU-bound задачи     | Блокируют Event Loop |
| Асинхронная сложность| Требует async-стиля |
| Совместимость        | Не все библиотеки async |

Типовые сценарии

| Сценарий                    | Почему asyncio |
|-----------------------------|----------------|
| Web-серверы                 | Масштабируемый I/O |
| API-клиенты                 | Много запросов |
| Чат-системы                 | Одновременные соединения |
| Фоновые сервисы             | Эффективное ожидание |

Ключевая идея
AsyncIO реализует кооперативную многозадачность:
задачи добровольно передают управление через await,
а Event Loop обеспечивает эффективное выполнение без блокировок.

### Многопоточность (threading) <a id="threading"></a> 

Многопоточность в Python — модель параллельного выполнения задач внутри одного процесса.

Модуль threading используется для конкурентного выполнения операций,
особенно эффективен для I/O-bound задач.

Назначение threading

| Задача                    | Описание |
|---------------------------|----------|
| I/O-bound операции        | Сеть, файлы, ожидание |
| Параллельные ожидания     | Одновременная работа задач |
| Реактивность              | Повышение отзывчивости приложений |
| Разделяемая память        | Совместный доступ к данным |

Основные концепции

| Понятие        | Суть |
|----------------|------|
| Thread         | Поток выполнения |
| GIL            | Глобальная блокировка интерпретатора |
| Shared Memory  | Общая память потоков |
| Lock           | Блокировка доступа |
| Race Condition | Состояние гонки |

Архитектура выполнения

| Элемент        | Описание |
|----------------|----------|
| Главный поток  | Управляет программой |
| Рабочие потоки | Выполняют задачи |
| Общие данные   | Доступны всем потокам |
| Синхронизация  | Предотвращает конфликты |

Механизмы синхронизации

| Механизм       | Назначение |
|----------------|------------|
| Lock           | Взаимное исключение |
| RLock          | Рекурсивная блокировка |
| Semaphore      | Ограничение доступа |
| Event          | Сигнализация между потоками |
| Condition      | Ожидание состояния |

Сравнение с другими моделями

| Модель           | Подходит для | Ограничения |
|------------------|--------------|-------------|
| threading        | I/O-bound    | Ограничен GIL |
| multiprocessing  | CPU-bound    | Большие накладные расходы |
| asyncio          | I/O-bound    | Асинхронная сложность |

Преимущества

| Плюс                | Описание |
|---------------------|----------|
| Простота            | Понятная модель |
| Низкие накладные расходы | Лёгкие потоки |
| Общая память        | Быстрый обмен данными |
| Поддержка библиотек| Совместимость с sync-кодом |

Недостатки

| Минус                 | Описание |
|-----------------------|----------|
| GIL                   | Нет реального параллелизма CPU |
| Race conditions       | Ошибки синхронизации |
| Deadlock              | Взаимная блокировка |
| Сложность отладки     | Недетерминированное поведение |

Типовые сценарии

| Сценарий                     | Почему threading |
|------------------------------|------------------|
| HTTP-запросы                 | Ожидание сети |
| Работа с файлами             | Блокирующий I/O |
| Фоновые задачи               | Неблокирующие операции |
| GUI-приложения               | Отзывчивость интерфейса |

Лучшие практики

| Практика                     | Зачем нужна |
|------------------------------|------------|
| Минимизировать общие данные  | Снижение гонок |
| Использовать Lock            | Контроль доступа |
| Делать потоки короткими      | Упрощение управления |
| Логировать поведение         | Упрощение отладки |

Ключевая идея
threading подходит для I/O-bound задач и повышения отзывчивости.
Он не обеспечивает параллельные вычисления CPU из-за GIL,
но остаётся важным инструментом конкурентного программирования в Python.

### Многопроцессность (multiprocessing) <a id="multiprocessing"></a>

Многопроцессность в Python — подход параллельного выполнения задач с использованием нескольких процессов.

Модуль multiprocessing позволяет обходить GIL и эффективно использовать несколько ядер CPU.

Назначение multiprocessing

| Задача                    | Описание |
|---------------------------|----------|
| CPU-bound задачи          | Вычислительно сложные операции |
| Параллельное выполнение   | Одновременный запуск задач |
| Масштабирование           | Использование всех ядер процессора |
| Изоляция                 | Разделение памяти между задачами |

Основные концепции

| Понятие        | Суть |
|----------------|------|
| Process        | Отдельный процесс ОС |
| GIL            | Ограничение потоков, отсутствует между процессами |
| IPC            | Межпроцессное взаимодействие |
| Queue / Pipe   | Передача данных между процессами |
| Pool           | Пул рабочих процессов |

Архитектура выполнения

| Элемент        | Описание |
|----------------|----------|
| Главный процесс| Управляет созданием процессов |
| Рабочие процессы| Выполняют задачи |
| Обмен данными  | Через IPC-механизмы |
| Синхронизация  | Блокировки и семафоры |

Механизмы взаимодействия процессов

| Механизм       | Назначение |
|----------------|------------|
| Queue          | Потокобезопасная очередь |
| Pipe           | Двусторонний канал |
| Manager        | Общие объекты |
| Shared Memory  | Общая память между процессами |
| Lock / Semaphore | Синхронизация доступа |

Сравнение с другими моделями

| Модель           | Подходит для | Ограничения |
|------------------|--------------|-------------|
| threading        | I/O-bound    | Ограничен GIL |
| asyncio          | I/O-bound    | Один поток CPU |
| multiprocessing  | CPU-bound    | Больше накладных расходов |

Преимущества

| Плюс                  | Описание |
|-----------------------|----------|
| Обход GIL             | Реальный параллелизм |
| Масштабируемость      | Использование всех ядер |
| Изоляция              | Ошибки не ломают весь процесс |
| Гибкость              | Разные модели IPC |

Недостатки

| Минус                 | Описание |
|-----------------------|----------|
| Высокие накладные расходы | Создание процессов |
| Сложность отладки     | Многопроцессная логика |
| Передача данных       | Медленнее, чем в потоках |
| Платформенные различия| Особенно Windows |

Типовые сценарии

| Сценарий                     | Почему multiprocessing |
|------------------------------|------------------------|
| Обработка изображений        | Интенсивные вычисления |
| Анализ больших данных        | Параллельные расчёты |
| Научные вычисления           | Использование CPU |
| Batch-обработка              | Независимые задачи |

Лучшие практики

| Практика                    | Зачем нужна |
|-----------------------------|------------|
| Использовать Pool            | Контроль количества процессов |
| Минимизировать IPC           | Снижение накладных расходов |
| Явная точка входа            | Совместимость с Windows |
| Делить задачи крупно         | Эффективная загрузка CPU |

Ключевая идея
multiprocessing позволяет выполнять CPU-нагруженные задачи параллельно, обходя ограничения GIL.
Он незаменим там, где требуется реальное использование многоядерных процессоров.

### Асинхронные генераторы и контекстные менеджеры <a id="async-generators"></a>

Асинхронные генераторы и асинхронные контекстные менеджеры — инструменты для управления потоками данных и ресурсами
в асинхронных приложениях Python.

Они используются вместе с asyncio и позволяют писать неблокирующий и читаемый код.

Асинхронные генераторы

Асинхронный генератор — функция, которая возвращает значения по частям в асинхронном контексте.

Назначение асинхронных генераторов

| Задача                    | Описание |
|---------------------------|----------|
| Потоковая обработка данных| Передача данных частями |
| Асинхронный I/O           | Работа с сетью и файлами |
| Экономия памяти           | Нет хранения всего набора данных |
| Композиция async-кода     | Интеграция с asyncio |

Ключевые особенности

| Особенность        | Суть |
|-------------------|------|
| async def          | Определение асинхронного генератора |
| yield              | Возврат значения |
| async for          | Итерация по значениям |
| await              | Асинхронные операции внутри |

Преимущества

| Плюс                 | Описание |
|----------------------|----------|
| Неблокирующее выполнение | Высокая масштабируемость |
| Ленивые вычисления   | Данные вычисляются по требованию |
| Читаемость           | Линейный стиль async-кода |

Асинхронные контекстные менеджеры

Асинхронный контекстный менеджер управляет ресурсами в асинхронной среде.

Назначение асинхронных контекстных менеджеров

| Задача                | Описание |
|-----------------------|----------|
| Управление ресурсами  | Соединения, сессии, блокировки |
| Гарантия освобождения | Корректное закрытие ресурсов |
| Читаемость            | Явные границы жизненного цикла |
| Обработка ошибок      | Безопасный выход при исключениях |

Ключевые особенности

| Особенность        | Суть |
|-------------------|------|
| async with         | Использование контекстного менеджера |
| __aenter__         | Асинхронный вход |
| __aexit__          | Асинхронный выход |
| await              | Асинхронные операции управления |

Сравнение

| Критерий              | Асинхронные генераторы | Асинхронные контекстные менеджеры |
|-----------------------|------------------------|-----------------------------------|
| Назначение            | Поток данных           | Управление ресурсами              |
| Модель использования  | async for              | async with                        |
| Основная задача       | Итерация               | Вход / выход                      |
| Тип абстракции        | Последовательность     | Жизненный цикл                    |

Типовые сценарии использования

| Сценарий                   | Инструмент |
|----------------------------|------------|
| Поток данных из API        | Async генератор |
| Чтение больших файлов     | Async генератор |
| Работа с БД соединением    | Async контекстный менеджер |
| HTTP-сессии                | Async контекстный менеджер |
| Блокировки и семафоры      | Async контекстный менеджер |

Ограничения

| Ограничение           | Описание |
|----------------------|----------|
| Сложность отладки     | Асинхронный стек сложнее |
| Требует asyncio      | Не работает в sync-коде |
| Совместимость        | Не все библиотеки async |

Ключевая идея
Асинхронные генераторы отвечают за потоковое получение данных без блокировок.
Асинхронные контекстные менеджеры отвечают за безопасное управление ресурсами в async-среде.
Вместе они позволяют писать эффективный и чистый асинхронный Python-код.

### Тестирование (unittest, pytest) <a id="тестирование"></a>

Тестирование в Python — процесс проверки корректности и стабильности кода с помощью автоматических тестов.

unittest и pytest — два основных фреймворка тестирования, применяемые для разных стилей и задач.

Назначение тестирования

| Цель                | Описание |
|---------------------|----------|
| Проверка логики     | Подтверждение корректной работы кода |
| Регрессия           | Защита от повторного появления ошибок |
| Рефакторинг         | Безопасные изменения кода |
| Документирование    | Тесты как описание поведения системы |
| CI/CD               | Автоматическая проверка качества |

Общие концепции тестирования

| Понятие        | Суть |
|----------------|------|
| Test case      | Отдельная проверка поведения |
| Assertion      | Утверждение ожидаемого результата |
| Fixture        | Подготовка и очистка окружения |
| Test runner    | Запуск и управление тестами |
| Mock           | Подмена зависимостей |

unittest — встроенный фреймворк тестирования.

Характеристика unittest

| Аспект              | Описание |
|---------------------|----------|
| Происхождение       | Стандартная библиотека |
| Стиль               | Классовый (xUnit) |
| Структура           | Жёстко формализована |
| Расширяемость       | Через модули и классы |
| Совместимость       | Хорош для legacy-кода |

pytest — современный фреймворк тестирования.

Характеристика pytest

| Аспект              | Описание |
|---------------------|----------|
| Происхождение       | Сторонняя библиотека |
| Стиль               | Функциональный |
| Синтаксис           | Минимум шаблонного кода |
| Расширяемость       | Плагины и фикстуры |
| Гибкость            | Высокая |

Сравнение unittest и pytest

| Критерий            | unittest              | pytest               |
|---------------------|-----------------------|----------------------|
| Входит в Python     | Да                    | Нет                  |
| Стиль               | Классы                | Функции и классы     |
| Читаемость          | Средняя               | Высокая              |
| Фикстуры            | Ограниченные          | Мощные и гибкие      |
| Параметризация      | Ограниченная          | Продвинутая          |
| Сообщество          | Стабильное            | Очень активное       |

Типы тестов

| Тип теста       | Назначение |
|-----------------|------------|
| Unit            | Проверка отдельных компонентов |
| Integration    | Проверка взаимодействия частей |
| Functional     | Проверка бизнес-логики |
| End-to-End     | Проверка всей системы целиком |

Лучшие практики

| Практика                | Зачем нужна |
|-------------------------|------------|
| Малые тесты             | Быстрое выполнение |
| Изоляция                | Независимость тестов |
| Явные проверки          | Предсказуемость |
| Автоматический запуск   | Раннее выявление ошибок |
| Покрытие критичной логики | Надёжность системы |

Ограничения тестирования

| Ограничение        | Описание |
|-------------------|----------|
| Не доказывает отсутствие ошибок | Только снижает риск |
| Требует поддержки  | Тесты тоже код |
| Ложные срабатывания| Неверные ожидания |

Ключевая идея
unittest обеспечивает строгую и формализованную структуру тестов.
pytest делает тестирование проще, гибче и выразительнее.
Выбор зависит от масштаба проекта, требований и стиля команды.

### Форматирование и линтинг (black, flake8) <a id="линтинг"></a>

Форматирование и линтинг — инструменты обеспечения качества и единообразия Python-кода.

Форматирование отвечает за внешний вид кода.
Линтинг отвечает за анализ кода и поиск потенциальных ошибок.

black — автоматический форматтер кода.

Назначение black

| Задача              | Описание |
|---------------------|----------|
| Автоформатирование  | Приведение к единому стилю |
| Устранение споров   | Нет ручных решений по стилю |
| Читаемость          | Предсказуемый внешний вид |
| Интеграция          | Использование в CI/CD и IDE |

Особенности black

| Характеристика      | Описание |
|---------------------|----------|
| Opinionated         | Жёстко заданный стиль |
| Без конфигурации    | Минимум настроек |
| PEP 8               | Совместим с рекомендациями |
| Детерминированность | Один и тот же результат всегда |

flake8 — инструмент статического анализа кода.

Назначение flake8

| Задача                | Описание |
|-----------------------|----------|
| Поиск ошибок          | Потенциальные баги |
| Проверка стиля        | Соответствие PEP 8 |
| Анализ сложности     | Выявление сложных участков |
| Контроль качества    | Поддержание стандартов |

Что проверяет flake8

| Проверка              | Суть |
|-----------------------|------|
| Синтаксис             | Ошибки Python |
| PEP 8                 | Стиль оформления |
| Неиспользуемый код    | Импорты, переменные |
| Сложность             | Чрезмерная вложенность |

Сравнение black и flake8

| Критерий        | black                  | flake8               | 
|-----------------|------------------------|----------------------|
| Тип инструмента | Форматтер              | Линтер               |
| Исправляет код  | Да                     | Нет                  |
| Анализ логики   | Нет                    | Частично             |
| Конфигурация    | Минимальная            | Гибкая               |
| Назначение      | Единый стиль           | Контроль ошибок      |

Совместное использование

| Подход              | Польза |
|---------------------|--------|
| black → flake8      | Форматирование перед анализом |
| CI/CD               | Автоматическая проверка |
| Pre-commit hooks    | Проверка до коммита |
| IDE интеграция      | Мгновенная обратная связь |

Лучшие практики

| Практика               | Зачем нужна |
|------------------------|------------|
| Форматировать всегда   | Устранение споров |
| Линтить регулярно      | Ранний поиск ошибок |
| Общие настройки        | Консистентность |
| Автоматизация          | Минимум ручной работы |

Ограничения

| Ограничение           | Описание |
|----------------------|----------|
| black не ищет ошибки  | Только форматирование |
| flake8 не исправляет  | Только анализ |
| Возможны конфликты    | Требует согласованных настроек |

Ключевая идея
black делает код единообразным.
flake8 контролирует качество и выявляет ошибки.
Вместе они формируют основу чистого и поддерживаемого Python-кода.

### Документирование и docstrings <a id="документирование"></a>

Документирование и docstrings — способ описания кода, его назначения и поведения прямо внутри Python-программы.

Документация делает код понятным, поддерживаемым и удобным для использования другими разработчиками и инструментами.

Назначение документирования

| Цель                | Описание |
|---------------------|----------|
| Понимание кода      | Объяснение логики и назначения |
| Поддержка           | Упрощение сопровождения |
| Использование API   | Пояснение интерфейсов функций и классов |
| Автогенерация docs  | Создание документации автоматически |

Docstring — строка документации, связанная с модулем, классом, функцией или методом.

Области применения docstrings

| Объект     | Назначение |
|------------|------------|
| Модуль     | Общее описание файла |
| Класс      | Назначение и поведение класса |
| Функция    | Описание логики и параметров |
| Метод      | Поведение метода класса |

Структура docstring

| Элемент      | Суть |
|--------------|------|
| Краткое описание | Что делает объект |
| Подробное описание | Как и зачем используется |
| Параметры    | Описание входных данных |
| Возвращаемое значение | Что возвращает |
| Исключения   | Возможные ошибки |

Популярные стили docstrings

| Стиль        | Характеристика |
|--------------|----------------|
| PEP 257      | Базовый стандарт Python |
| Google style | Краткий и читаемый |
| NumPy style  | Подробный, для научных библиотек |
| reStructuredText | Используется Sphinx |

Инструменты, использующие docstrings

| Инструмент | Назначение |
|-----------|------------|
| help      | Встроенная справка |
| IDE       | Подсказки и автодополнение |
| Sphinx   | Генерация документации |
| pydoc    | Просмотр документации |
| MkDocs   | Документация проектов |

Лучшие практики

| Практика                 | Зачем нужна |
|--------------------------|------------|
| Краткость                | Быстрое понимание |
| Единый стиль             | Консистентность проекта |
| Актуальность             | Соответствие коду |
| Документирование API     | Удобство для пользователей |
| Описание сложной логики  | Снижение порога входа |

Что стоит документировать

| Объект                    | Причина |
|---------------------------|---------|
| Публичные функции         | Используются извне |
| Классы и интерфейсы       | Основа архитектуры |
| Модули                    | Контекст и назначение |
| Нетипичную логику         | Сложность понимания |

Частые ошибки

| Ошибка                  | Последствие |
|-------------------------|-------------|
| Отсутствие docstrings   | Непонятный код |
| Дублирование очевидного | Захламление документации |
| Устаревшие описания     | Дезинформация |
| Несогласованный стиль   | Плохая читаемость |

Ключевая идея
Docstrings — это встроенная документация Python-кода.
Они служат связующим звеном между кодом, разработчиком и инструментами автоматической генерации документации.
Хорошая документация повышает качество, поддерживаемость и ценность проекта.

### Сборка и публикация пакетов <a id="публикация-пакетов"></a>

Сборка и публикация пакетов — процесс подготовки Python-проекта к распространению и установке другими пользователями.

Цель — превратить исходный код в стандартный дистрибутив и разместить его в репозитории пакетов.

Основные этапы

| Этап                | Назначение |
|---------------------|------------|
| Подготовка проекта  | Описание метаданных и структуры |
| Сборка              | Создание дистрибутивов |
| Проверка            | Валидация пакета |
| Публикация          | Загрузка в репозиторий |
| Установка           | Использование конечными пользователями |

Форматы дистрибутивов

| Формат | Назначение |
|-------|------------|
| sdist | Исходный архив пакета |
| wheel | Предсобранный бинарный пакет |

Инструменты сборки

| Инструмент    | Роль |
|---------------|------|
| setuptools    | Описание и сборка пакета |
| wheel         | Создание wheel-дистрибутива |
| build         | Унифицированный инструмент сборки |
| Poetry        | Сборка и публикация «из коробки» |

Метаданные пакета

| Метаданные        | Назначение |
|-------------------|------------|
| Имя               | Уникальный идентификатор пакета |
| Версия            | Управление релизами |
| Описание          | Назначение библиотеки |
| Зависимости       | Требуемые библиотеки |
| Лицензия          | Правила использования |

Репозитории пакетов

| Репозиторий | Назначение |
|------------|------------|
| PyPI       | Основной публичный репозиторий |
| TestPyPI   | Тестовая публикация |
| Private    | Внутренние репозитории компаний |

Проверка перед публикацией

| Проверка            | Зачем нужна |
|---------------------|------------|
| Корректность версии | Избежание конфликтов |
| Совместимость       | Поддержка Python версий |
| Зависимости         | Отсутствие лишних пакетов |
| Документация        | Понимание назначения пакета |

Автоматизация процесса

| Подход        | Польза |
|---------------|--------|
| CI/CD         | Автоматическая сборка |
| Теги Git      | Привязка версий к релизам |
| Semantic Versioning | Предсказуемые обновления |
| Проверки      | Минимизация ошибок |

Типовые ошибки

| Ошибка                  | Последствие |
|-------------------------|-------------|
| Отсутствие wheel        | Медленная установка |
| Неполные метаданные     | Плохая читаемость пакета |
| Сломанные зависимости   | Ошибки установки |
| Ручная публикация       | Человеческий фактор |

Когда публиковать пакет

| Ситуация                    | Причина |
|-----------------------------|---------|
| Переиспользуемый код        | Повторное использование |
| Open Source проект          | Доступность сообщества |

| CLI-инструменты             | Удобная установка |

Ключевая идея
Сборка и публикация пакетов превращают Python-проект в стандартный, устанавливаемый и распространяемый продукт.
Корректная сборка, фиксация версий и автоматизация делают процесс надёжным и воспроизводимым.
